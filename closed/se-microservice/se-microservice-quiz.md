## 题目

### Q1 [选择题] 微服务架构中，防止故障级联的核心机制是？

A. 分布式事务（2PC）
B. 熔断器（Circuit Breaker）
C. 配置中心（Nacos）
D. API 网关

- **正确答案**：B
- **评分要点**：选择正确选项即通过
- **知识点**：熔断器

### Q2 [选择题] 以下哪种情况说明服务拆得太细了？

A. 每个服务有独立的数据库
B. 改一个功能经常需要同时改三个服务的代码
C. 不同服务使用不同的编程语言
D. API 网关需要聚合多个服务的数据

- **正确答案**：B
- **评分要点**：选择正确选项即通过。注意 ACD 都是微服务的正常特征
- **知识点**：服务拆分

### Q3 [简答题] 解释 Saga 模式的工作原理。以下单流程（订单→库存→支付）为例，如果支付失败，整个流程如何回滚？

- **参考答案**：
  1. 订单服务创建订单 → 发事件
  2. 库存服务扣库存 → 发事件
  3. 支付服务扣款失败 → 发补偿事件
  4. 库存服务收到补偿 → 加回库存
  5. 订单服务收到补偿 → 取消订单
  每个"回滚"不是数据库回滚，而是执行一个相反的操作（补偿事件）
- **评分要点**：
  1. 必须提到补偿事件/反向操作
  2. 必须说明是逐步递推回滚，不是一次性回滚
  3. 加分项：提到最终一致性
- **知识点**：Saga 模式

### Q4 [简答题] 为什么微服务中每个服务要有独立数据库？共享数据库的核心问题是什么？

- **参考答案**：
  1. 共享数据库 = 共享数据 = 耦合
  2. 两个服务写同一张表，改表结构要协调多个团队，失去独立部署的意义
  3. 锁竞争导致性能下降
  4. 独立数据库才能实现真正的服务自治
- **评分要点**：
  1. 必须提到"耦合"这个核心问题（不仅是锁竞争）
  2. 必须提到"改表结构协调多团队"
  3. 加分项：提到共享数据库 = 失去独立部署意义
- **知识点**：独立数据库

### Q5 [简答题] 幂等性在微服务中为什么特别重要？举一个非幂等操作导致问题的具体场景。

- **参考答案**：
  网络不可靠导致重试。服务 A 调服务 B 扣款 100 元，B 处理完了但响应丢了，A 以为失败又发一次，B 再扣 100 元。如果用"设置余额为 X"（幂等）而不是"扣减 100 元"（非幂等），或者带唯一请求 ID 去重，就不会重复扣款。
- **评分要点**：
  1. 必须说明网络不可靠导致重试
  2. 必须举出具体场景（扣款、下单等）
  3. 加分项：提到至少一种幂等实现方式
- **知识点**：幂等性

### Q6 [对比题] 比较同步调用和异步消息在微服务通信中的优缺点。

- **参考答案**：
  | 维度 | 同步调用（REST/gRPC） | 异步消息（Kafka/RabbitMQ） |
  |------|----------------------|--------------------------|
  | 响应 | 即时返回结果 | 发完就走，不等待 |
  | 耦合 | 服务间强依赖 | 完全解耦 |
  | 故障 | 调用链级联（雪崩） | 上游不受下游影响 |
  | 调试 | 简单（同步调用栈） | 困难（消息丢失/重复/乱序） |
  | 场景 | 用户在等结果 | 后台任务、削峰填谷 |
- **评分要点**：至少指出 3 个对比维度
- **知识点**：服务间通信

### Q7 [对比题] 环境变量和配置中心各适合什么场景？配置中心挂了怎么办？

- **参考答案**：
  | 维度 | 环境变量 | 配置中心 |
  |------|---------|---------|
  | 更新方式 | 改了要重启 | 热更新，不用重启 |
  | 管理方式 | 散落各服务 | 集中管理 |
  | 版本管理 | 无 | 支持版本和灰度发布 |
  | 适合场景 | 端口、日志级别等不常变的 | 功能开关、阈值等常变的 |
  配置中心挂了：各服务用本地缓存继续跑，不影响运行
- **评分要点**：至少指出 2 个区别维度 + 提到容灾方案
- **知识点**：配置管理

### Q8 [应用题] 你负责设计一个外卖平台的微服务架构。请列出你会拆分哪些服务，并说明"用户下单→骑手接单"这个流程中，哪些步骤用同步通信、哪些用异步通信，为什么？

- **参考答案**：
  拆分建议：订单服务、用户服务、商家服务、支付服务、配送服务
  - 下单（同步）：用户等着看订单是否创建成功
  - 扣款（同步）：用户等着看支付结果
  - 通知商家（异步）：商家后台收到新订单提醒，用户不需要等商家确认
  - 分配骑手（异步）：系统后台匹配骑手，用户不需要实时等
  - 骑手接单（异步）：骑手接单后推送通知给用户
- **评分要点**：
  1. 服务拆分合理（按业务能力）
  2. 下单和扣款判断为同步
  3. 通知和配送判断为异步
  4. 能说出判断理由
- **知识点**：服务拆分 + 通信方式综合

### Q9 [应用题] 你的系统有 A→B→C 三个服务调用链。最近 C 服务经常超时，导致 B 的线程池耗尽，A 也被拖死。你会用什么方案解决？具体怎么做？

- **参考答案**：
  1. **熔断器**：在 B 调 C 的地方加熔断器。C 连续超时超过阈值，熔断器打开，B 直接返回错误不再调 C，保护 B 不被拖死
  2. **舱壁隔离**：给 B 调 C 分配独立线程池，C 出问题只影响这个池子，B 调其他服务的线程不受影响
  3. **超时设置**：给 B 调 C 设合理的超时时间，不要无限等待
  三者配合使用，层层防护
- **评分要点**：
  1. 必须提到熔断器
  2. 加分项：提到舱壁隔离
  3. 加分项：提到超时设置
- **知识点**：熔断器 + 舱壁隔离

### Q10 [简答题] API 网关有哪些核心职责？BFF 和普通网关的区别是什么？

- **参考答案**：
  核心职责（路认限聚）：
  1. 路由：把请求转发到对应服务
  2. 认证：统一验证 token
  3. 限流：防止客户端打爆服务
  4. 聚合：组合多个服务的数据返回给前端
  BFF 区别：普通网关一个入口服务所有客户端，BFF 给每种客户端一个专属网关（如 App 一个、网页一个），避免单个网关承载所有逻辑变得臃肿
- **评分要点**：
  1. 四个职责至少提 3 个
  2. 必须提到 BFF 的区别
- **知识点**：API 网关

## 题目

### 第 1 题（选择题）

**关于 In-Context Learning (ICL)，以下哪个说法是正确的？**

A) ICL 通过在 prompt 中提供示例，微调模型参数以适应新任务
B) ICL 不更新模型参数，其效果可能来自隐式梯度下降或模式匹配机制
C) ICL 的效果主要依赖模型规模，与预训练数据中的模式无关
D) ICL 和 Few-Shot 是完全不同的技术，没有关联

**答案：B**
- A 错在 ICL 不更新参数（那是 fine-tuning）
- C 错在预训练数据中的模式是根本原因
- D 错在 Few-Shot 是 ICL 的一种具体实现

---

### 第 2 题（选择题）

**关于 Chain-of-Thought (CoT)，以下哪个说法最准确？**

A) CoT 让模型在输出前展示推理过程，本质上是让模型变得更"聪明"
B) CoT 的效果完全来自模型规模增大后的涌现能力
C) CoT 把复杂推理拆成多步，每生成一个中间 token 等于多一次前向传播，增加了有效计算量
D) CoT 对所有任务都有正面效果，应该始终使用

**答案：C**
- A 错在不是变"聪明"，而是给更多计算轮次
- B 部分对但不全面
- D 错在简单任务用 CoT 可能过度推理

### 第 3 题（简答题）

**Few-Shot Prompting 的核心价值是什么？为什么说它不是在"教模型新知识"？**

**答案要点：**
- 核心价值：消除任务歧义 + 激活预训练模式匹配能力
- 不是教新知识，而是告诉模型"你想要什么格式"
- 示例触发模型预训练中已学到的模式（教科书例题→习题模式）

### 第 4 题（简答题）

**Constrained Decoding 中，有限状态机（FSM）的作用是什么？它和普通的 JSON 语法检查有什么区别？**

**答案要点：**
- FSM 将 JSON Schema 编译成状态机，追踪当前走到 Schema 的哪个位置
- JSON 语法检查只管括号配对、引号闭合（是否合法 JSON）
- FSM 还管字段名、值类型、必填项（是否符合具体 Schema）
- 两个层次：合法 JSON vs 符合 Schema 的 JSON

### 第 5 题（简答题）

**System Prompt 和 User Prompt 在模型内部有什么实质性的区别？根本原因是什么？**

**答案要点：**
- 三机制：特殊 token 标记、注意力权重差异、训练数据结构
- 根本原因是训练数据结构——预训练/微调数据中 system-like 指令和用户查询有明确分隔
- 注意力权重差异和特殊 token 标记是训练的结果，不是原因

### 第 6 题（对比题）

**In-Context Learning (ICL) 和 Fine-Tuning 都能让模型学会新任务，请比较区别。**

**答案要点：**

| 维度 | ICL | Fine-Tuning |
|------|-----|-------------|
| 参数 | 不更新，效果临时 | 更新参数，效果永久 |
| 成本 | 在推理时（占上下文窗口） | 在训练时（需要算力） |
| 数据需求 | 几个示例 | 几十到几千条标注数据 |
| 速度 | 即时生效 | 需要训练时间 |
| 适用场景 | 快速实验、任务多变 | 固定任务、需要最高质量 |

### 第 7 题（对比题）

**第 1 代"在 Prompt 里说输出 JSON"和第 3 代 Constrained Decoding，本质区别是什么？**

**答案要点：**
- 第 1 代：软引导，通过 prompt 激活模式匹配（~80-95%）
- 第 3 代：硬约束，物理屏蔽非法 token 的 logit（~100%）
- 本质区别：请求 vs 强制
- 第 3 代只能保证格式正确，不能保证内容正确

### 第 8 题（应用题）

**设计一个客服机器人 prompt，将用户投诉分类为结构化 JSON 数据。用到哪些 Prompt Engineering 技巧？**

**答案要点：**
- System Prompt：定义角色 + 输出格式
- Few-Shot：提供多样化示例（覆盖不同分类和情感）
- Constrained Decoding：硬约束保证 JSON 合法
- 改进点：示例应覆盖不同类型（不只是负面），明确 urgency 数值含义

---

### 第 9 题（应用题）

**为什么 Few-Shot + CoT 组合使用比单独用 Few-Shot 效果更好？**

**答案要点：**
- Few-Shot 解决格式歧义（输出对不对）
- CoT 解决推理质量（答案对不对）
- Few-Shot CoT：示例中包含推理过程，教会模型"怎么想"而不只是"怎么答"
- 实践中 CoT + JSON 的冲突：可用 reasoning 字段、分离输出通道、两步调用等方案

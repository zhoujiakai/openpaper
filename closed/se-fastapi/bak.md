## 容易混淆的点（模拟题出错部分）

### 1. 认证用依赖注入，不用中间件

我在应用题里把认证写成了中间件，这是错的。

原因：不是所有路由都需要认证。`/public` 不用认证，`/me` 需要。如果放中间件里，就得自己写 if-else 判断路径，代码很快变乱。放在依赖注入里，哪个路由需要就加一行 `Depends(get_current_user)`，干净得多。

判断标准：
- **每个请求都要做**（日志、CORS、压缩）→ 中间件
- **只有部分路由需要**（认证、查数据库、分页）→ 依赖注入

### 2. lifespan 和 yield 依赖的边界

我在机制选择题里把两者搞混了。

核心区别就一个问题：**这个资源是所有人共享一份，还是每个请求各要各的？**

| 资源 | 级别 | 机制 | 原因 |
|------|------|------|------|
| 数据库连接池 | 应用级 | lifespan | 启动时创建一次，所有请求共享 |
| ML 模型 | 应用级 | lifespan | 加载到内存，所有人共用 |
| Redis 连接 | 应用级 | lifespan | 全局共享 |
| 数据库会话 | 请求级 | yield 依赖 | 每个请求借一个，用完还回去 |
| 数据库事务 | 请求级 | yield 依赖 | 每个请求独立开启，成功提交失败回滚 |
| 当前用户信息 | 请求级 | 普通 return 依赖 | 解析 token 就行，不需要清理 |

注意：当前用户信息虽然是请求级的，但它不需要清理，所以是普通的 `return` 依赖，不是 `yield` 依赖。

### 3. 请求/响应模型为什么要分两个类

我答了"字段不一定相同"，方向对但不够具体。三个具体区别：

1. **请求有，响应没有** — `password`，客户端要传进来存数据库，但绝不返回给前端
2. **响应有，请求没有** — `id`，服务端自动生成，客户端创建时不需要传
3. **字段类型可能不同** — 比如请求里 `password` 是明文字符串，响应里压根不该有这个字段

如果用同一个类，要么前端能拿到密码，要么前端没法传密码。

### 4. A4 默写出错的部分

**依赖注入 ≠ yield**

我一开始把依赖注入等同于 yield，这是错的。依赖注入是更大的概念：

- **依赖注入** = 用 `Depends()` 告诉框架"我需要什么"，框架帮你准备好传进来
- **`yield` 依赖** = 只是依赖注入的一种写法，专门处理需要清理的资源

大多数依赖是普通 `return` 的（解析 token、查配置、构造分页参数），只有数据库连接、文件句柄这些需要"借还"的资源才用 `yield`。

**中间件执行顺序**

A4 默写时我说"越靠近接口（最内层）的中间件会越先执行"，说反了。

正确：最后注册的中间件在最外层，最先接触请求。

```
注册顺序：A → B → C
请求经过：C → B → A → 路由函数 → A → B → C
```

### 5. lifespan vs on_event 的三个优势

A4 默写时只知道 lifespan 做什么，不知道和 on_event 的区别。三个优势：

1. **变量天然共享** — 一个函数管启动和关闭，不需要全局变量
2. **关闭逻辑保证执行** — yield 后面相当于 finally，不管有没有异常都会跑
3. **ASGI 标准协议** — 不依赖 FastAPI 特有功能，on_event 已废弃

## 追问深挖的内容

### FastAPI 用几个线程处理异步请求？

默认 **1 个线程**。一个 Uvicorn worker 就是一个进程，进程里跑一个事件循环（一个线程），所有异步路由都在这个事件循环里执行。靠异步 I/O 不阻塞，一个线程就能处理几百上千个并发。

生产环境开多个 worker（进程），每个 worker 各自一个事件循环：

```
nginx
  ├── uvicorn worker 1（1 个线程，1 个事件循环）
  ├── uvicorn worker 2（1 个线程，1 个事件循环）
  └── ...
```

### 同步路由用多少线程？

FastAPI 用固定大小的线程池，**默认 40 个线程**。不是每个请求创建一个线程，而是从池子里取。100 个并发同步请求只有 40 个能同时处理，剩下 60 个排队等。

| 路由类型 | 线程模型 | 并发上限 |
|----------|---------|---------|
| `async def` | 1 个事件循环线程 | 很高（几百上千） |
| `def` | 40 个线程池线程 | 受线程池大小限制 |

### 中间件注册顺序的实际例子

三个中间件的正确注册顺序：

```python
app.add_middleware(AuthMiddleware)      # 第一注册 = 最内层
app.add_middleware(CORSMiddleware)      # 第二注册 = 中间层
app.add_middleware(LoggingMiddleware)   # 第三注册 = 最外层
```

请求经过：Logging → CORS → Auth → 路由函数

为什么这么排：
- **Logging 最外层**（最后注册）— 包住所有中间件的耗时，记录完整时间
- **CORS 中间层** — 预检请求（OPTIONS，不带 token）先进 CORS 处理，不会被 Auth 拦截返回 401
- **Auth 最内层**（最先注册）— 只管需要认证的路由

### WSGI vs ASGI

- **WSGI** = Web Server Gateway Interface（同步标准）
- **ASGI** = Asynchronous Server Gateway Interface（异步标准）

就多了一个 A（Asynchronous）。Flask 走 WSGI（Gunicorn），FastAPI 走 ASGI（Uvicorn）。

### Flask 有没有异步？

Flask 2.0+ 加了有限的 async 支持，可以写 `async def` 视图。但底层仍是 WSGI，每个 async 请求还是占一个线程。不是原生异步。

## 关键概念速记

### 口诀

- 依赖注入：**"我需要什么，框架替我准备"** — Depends() 声明需求
- yield 依赖：**"借了要还"** — yield 前借出，finally 后归还
- 中间件：**"洋葱模型"** — 最后注册 = 最外层 = 最先进请求
- lifespan：**"开业租门，关门退租"** — 应用级资源，启动创建、关闭释放
- 异步：**"等 I/O 不闲着"** — 挂起协程去干别的

### 易混淆对比

| 概念 A | 概念 B | 区别 |
|--------|--------|------|
| 中间件 | 依赖注入 | 中间件全局拦截，依赖注入按路由按需 |
| lifespan | yield 依赖 | lifespan 管应用级共享资源，yield 依赖管请求级借用资源 |
| `async def` 路由 | `def` 路由 | async 在事件循环跑（配异步库），def 在线程池跑（同步库安全） |
| `yield` 依赖 | `return` 依赖 | yield 能做清理（借还），return 只给不管还 |
| `return` 依赖 | `yield` 依赖 | 不需要清理的资源用 return（如当前用户），需要清理的用 yield（如 db session） |

## 我的理解

FastAPI 的核心就是一条流水线：请求进来，经过中间件（质检）、依赖注入（领配件）、路由函数（加工）、响应模型（包装），最后出去。

- **中间件**管全局的事，每个请求必过，像保安
- **依赖注入**管局部的事，按路由声明需要什么，像领工具
- **lifespan** 管应用启停的事，创建和销毁全局共享资源
- **Pydantic** 管数据格式的事，进来验货，出去过滤
- **异步**管性能的事，等 I/O 时不闲着

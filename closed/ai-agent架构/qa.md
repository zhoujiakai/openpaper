# Agent 与工作流 — QA

> 学习过程中困惑的知识点
> 创建日期：2026-04-29

---

## Q1：Workflow vs Agent 的核心判断标准

**路径是代码定义的还是 LLM 动态决定的？**

- **Workflow**：路径由代码提前定义。LLM 在路径上的某个节点干活，但走哪条路是代码说了算。即使有循环（如 Evaluator-Optimizer），循环结构也是代码写死的。
- **Agent**：LLM 自己决定下一步做什么。每一步都可能改变方向，决策权在 LLM 手里。

**Orchestrator-Worker 是 Workflow 不是 Agent**

编排者只做一次决策："拆成几个子任务，分别做什么"。拆完后路径固定（工人执行 → 汇总 → 输出）。不像 Agent 那样在整个过程中持续决策。

**Evaluator-Optimizer 是 Workflow 不是 Agent**

循环结构是代码写死的（生成 → 评估 → 不通过再来一轮）。LLM 不决定"要不要循环"，代码决定。不像 ReAct 那样每步的行动类型都不确定。

## Q2：五种 Workflow 模式口诀

**"串路并编评"**（按路径复杂度递增）

- **串**行（Prompt Chaining）：串行分步，前一步输出是后一步输入
- **路**由（Routing）：先分类再分发，不同输入走不同处理流程。**不是规则引擎**，而是按输入类别分流。典型例子：客服系统中根据用户问题主题选择对应的处理流程
- **并**行（Parallelization）：子任务同时执行，最后汇总（或多次投票取多数）
- **编**排（Orchestrator-Worker）：包工头动态拆任务，分给工人做，最后汇总
- **评**估（Evaluator-Optimizer）：生成 → 评估 → 不通过就给反馈重做，循环到满意

## Q3：选型决策框架完整路径

从简到繁 8 步，第 2-6 步和"串路并编评"一一对应：

1. 单次 LLM 调用能解决？ → 直接用
2. 能拆成固定步骤？ → Prompt Chaining
3. 不同输入需要不同处理？ → Routing
4. 子任务之间独立？ → Parallelization
5. 子任务不确定，需要动态分配？ → Orchestrator-Worker
6. 结果需要反复打磨？ → Evaluator-Optimizer
7. 路径完全无法预测？ → Agent（ReAct 或 Plan-and-Execute）
8. 单 Agent 搞不定？ → Multi-Agent

**注意：** 第 7 步 Agent 有两种模式——ReAct（走一步看一步）和 Plan-and-Execute（先想好再走），适合不同复杂度的任务。

## Q4：Plan-and-Execute 的计划可以改

Plan-and-Execute 的计划并不是不能更改。如果执行到某一步发现前提错了，可以回去让规划者重新出方案。它的灵活性不是"不能改"，而是**出错才改**，不像 ReAct 每一步都在调整。

---

## Q5：ReAct 中 Thought 步骤为什么不能省

实验证明，去掉 Thought 只保留 Action，准确率会**明显下降**。原因：生成 Thought 的过程给了 LLM 更多上下文来做出更好的决策，减少盲目行动。这不是笼统的"避免无脑操作"，而是有实验数据支撑的结论，面试时可以直接甩出来。

## Q6：Multi-Agent 为什么 Supervisor 模式最常用

实际中 LangGraph 的 Send API 就是实现 Supervisor 模式的标准方式。选择 Supervisor 的原因：

1. **可控性强**：相比于 Hierarchical 模式，复杂度更低
2. **可追踪性**：相比于 Swarm 模式，Supervisor 有中心节点负责决策，出错了能定位。Swarm 是去中心化的，Agent 之间随意交接，出了错很难追踪是哪个 Agent 的责任，生产环境可靠性不够

## Q7：工具设计的具体教训（来自 Anthropic SWE-bench 实践）

**反直觉结论：花在工具设计上的时间比花在 Prompt 上的时间还多。** LLM 调用工具的本质是"读描述 → 决定怎么用"，工具描述和参数设计对 Agent 表现的影响比 Prompt 更大。

两个具体教训：

1. **相对路径 vs 绝对路径**：Agent 移动目录后会搞混相对路径，改成只接受绝对路径后错误消失
2. **写 diff 比重写整个文件更容易出错**：diff 需要精确计算行号，LLM 经常算错；让 LLM 在 JSON 里写代码比在 Markdown 里写代码也更容易出错（需要转义引号和换行）

三条工具设计原则：
- 格式要接近 LLM 在网上见过的自然格式
- 参数设计要防呆（让错误用法变得不可能）
- 像给初级开发写文档一样写工具描述（包括示例、边界情况）

## Q8：必须用 Agent 而不能用 Workflow 的真实场景

**场景：修复 GitHub Issue**

**为什么 Workflow 搞不定？**
- 不知道要改几个文件——可能是 1 个，可能是 5 个
- 不知道路径——先看哪里、再查什么，取决于每一步的发现
- 中间结果不可预测——读了一个文件后发现不是它的问题，要换方向
- 每一步的行动类型都不确定——可能要读代码、搜索关键词、运行测试、查看日志

**判断口诀：** 如果你能提前画出完整的流程图，用 Workflow；如果你画不出来（因为路径取决于中间结果），就用 Agent。

**补充理解：** 外层流程可以固定成 Workflow（搜索 → 修改 → 检查 → 提交），但每一步内部的执行路径完全不可预测，需要 Agent 自主决策。所以本质是"Workflow 外壳 + Agent 内核"。**关键不是外层框架长什么样，而是执行过程中 LLM 是否需要持续做不可预测的决策。**

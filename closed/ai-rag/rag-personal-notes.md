## RAG 分块策略选择原则

**固定大小 + Overlap 是最常用的通用 Baseline 方案。** 学术研究发现固定大小分块在很多基准测试中表现不亚于语义分块，不需要一味追求更复杂的策略。

**核心原则：固定大小 + Overlap 是基线，围绕数据的天然语义单元来设计分块。** 比如有明确章节结构的文档用结构感知分块，短文本用句子分块。

### 语义分块 vs LLM 驱动分块

两者都跟"语义"有关，但实现方式完全不同：

- **语义分块**：用 Embedding 模型计算相邻句子的向量相似度，相似度低于阈值就切一刀。是数学计算，成本低，但只能看相邻句子的局部相似度
- **LLM 驱动分块**：直接让 LLM 读文本决定在哪里切。能理解全文语境、逻辑结构、主题转换，判断力最强，但成本高、延迟大

一句话总结：语义分块是用数值（向量距离）判断，LLM 驱动分块是用语言理解能力判断。

## LangChain RAG 流水线五步

**Load → Split → Store → Retrieve → Generate**（加载 → 分割 → 存储 → 检索 → 生成）

**注意：** Reranking 不是独立步骤，是 Retrieve 的可选增强（粗筛 + 精排）。

**记忆方法：** 和 RAG 四阶段的映射关系——
- Load + Split + Store = Ingestion（索引阶段）
- Retrieve = Retrieval（检索阶段）
- Generate = Augmentation + Generation（增强 + 生成阶段）

## 混合搜索与 Reranking

### 纯向量检索（语义搜索）的局限

- 对专有名词、缩写、产品名称、错误码等精确匹配场景检索不准
- 比如 user 搜 "HTTP 404"，语义搜索可能返回"常见网络错误"的文档，而不是精确包含 "404" 的那段
- 所以需要混合搜索：语义搜索捕捉"意思接近"，关键词搜索捕捉"精确匹配"

### Reranking 的两种模型架构

**不是 Dense Search 和 Sparse Search**（那是两种检索方式），而是：

- **Bi-encoder（双塔模型）**：query 和 doc 各自独立编码成向量，算余弦相似度。速度快但精度有限（两个文本不互相看）
- **Cross-encoder（交叉编码器）**：query 和 doc 拼在一起同时编码。精度高（能看两者的交互），但速度慢

**为什么不能直接用 Cross-encoder？** 因为要把 query 和每个候选文档拼一起过一遍模型，100 万条文档就要跑 100 万次，太慢。所以先用 Bi-encoder 快速筛出 top-100，再用 Cross-encoder 对这 100 个精排，选出最终 top-10。

## RAG / 微调 / 长上下文 — 解决的是不同维度的问题

- **RAG** 解决的是**知识**问题——模型知道什么（适合企业内部知识库）
- **微调** 解决的是**行为**问题——模型怎么输出（适合固定格式输出，如企业文档生成）
- **长上下文** 解决的是**全局**问题——能不能看到完整信息（适合对单篇长文档的全局理解）

三者天然可以组合。经典例子：用 RAG 提供企业知识 + 用微调控制输出格式，生成企业内部报告。

## RAG 生成偏离排查

检索到了正确内容但生成偏离，从检索端和生成端两个方向排查：

**检索端：**
- top-k 值设太大，混入不相关噪声分块干扰 LLM

**生成端：**
- Prompt 约束不够强（加"不知道就说不知道"减少幻觉）
- **模型先验干扰**：LLM 自身训练数据里有相关知识，会"自信"地用自己的知识覆盖检索内容
- **Few-shot 示例不当**：示例的风格或内容引导 LLM 偏离

**排查三因口诀：Prompt 不够强、top-k 有噪声、模型先验干扰。**

## Embedding 模型选择维度

选 Embedding 模型就是挑"翻译器"（把文本变成向量），看四件事：

1. **向量维度**：越长越精细但越贵越慢。看预算和数据量选
2. **多语言支持**：中文知识库要选支持中文的（如 BGE 系列），OpenAI 的也行但成本更高
3. **领域特化**：医疗、法律等特殊领域有专门训练过的模型，效果更好
4. **跑分（MTEB 排行榜）**：选模型前去 MTEB 看看，同价位里挑分高的

常见模型：OpenAI text-embedding（通用）、BGE 系列（开源，中文友好）、E5 系列（微软，多语言）

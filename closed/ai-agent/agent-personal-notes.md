> 记录学习过程中觉得重要的内容
> 创建日期：2026-04-29

## Workflow vs Agent 的核心判断标准

**路径是代码定义的还是 LLM 动态决定的？**

- **Workflow**：路径由代码提前定义。LLM 在路径上的某个节点干活，但走哪条路是代码说了算。即使有循环（如 Evaluator-Optimizer），循环结构也是代码写死的。
- **Agent**：LLM 自己决定下一步做什么。每一步都可能改变方向，决策权在 LLM 手里。

### Orchestrator-Worker 是 Workflow 不是 Agent

编排者只做一次决策："拆成几个子任务，分别做什么"。拆完后路径固定（工人执行 → 汇总 → 输出）。不像 Agent 那样在整个过程中持续决策。

### Evaluator-Optimizer 是 Workflow 不是 Agent

循环结构是代码写死的（生成 → 评估 → 不通过再来一轮）。LLM 不决定"要不要循环"，代码决定。不像 ReAct 那样每步的行动类型都不确定。

---

## 五种 Workflow 模式口诀

**"串路并编评"**（按路径复杂度递增）

- **串**行（Prompt Chaining）：串行分步，前一步输出是后一步输入
- **路**由（Routing）：先分类再分发，不同输入走不同处理流程。**不是规则引擎**，而是按输入类别分流。典型例子：客服系统中根据用户问题主题选择对应的处理流程
- **并**行（Parallelization）：子任务同时执行，最后汇总（或多次投票取多数）
- **编**排（Orchestrator-Worker）：包工头动态拆任务，分给工人做，最后汇总
- **评**估（Evaluator-Optimizer）：生成 → 评估 → 不通过就给反馈重做，循环到满意

---

## 选型决策框架完整路径

从简到繁 8 步，第 2-6 步和"串路并编评"一一对应：

1. 单次 LLM 调用能解决？ → 直接用
2. 能拆成固定步骤？ → Prompt Chaining
3. 不同输入需要不同处理？ → Routing
4. 子任务之间独立？ → Parallelization
5. 子任务不确定，需要动态分配？ → Orchestrator-Worker
6. 结果需要反复打磨？ → Evaluator-Optimizer
7. 路径完全无法预测？ → Agent（ReAct 或 Plan-and-Execute）
8. 单 Agent 搞不定？ → Multi-Agent

**注意：** 第 7 步 Agent 有两种模式——ReAct（走一步看一步）和 Plan-and-Execute（先想好再走），适合不同复杂度的任务。

---

## Plan-and-Execute 的计划可以改

Plan-and-Execute 的计划并不是不能更改。如果执行到某一步发现前提错了，可以回去让规划者重新出方案。它的灵活性不是"不能改"，而是**出错才改**，不像 ReAct 每一步都在调整。

---

## ReAct 中 Thought 步骤为什么不能省

实验证明，去掉 Thought 只保留 Action，准确率会**明显下降**。原因：生成 Thought 的过程给了 LLM 更多上下文来做出更好的决策，减少盲目行动。这不是笼统的"避免无脑操作"，而是有实验数据支撑的结论，面试时可以直接甩出来。

---

## Multi-Agent 为什么 Supervisor 模式最常用

实际中 LangGraph 的 Send API 就是实现 Supervisor 模式的标准方式。选择 Supervisor 的原因：

1. **可控性强**：相比于 Hierarchical 模式，复杂度更低
2. **可追踪性**：相比于 Swarm 模式，Supervisor 有中心节点负责决策，出错了能定位。Swarm 是去中心化的，Agent 之间随意交接，出了错很难追踪是哪个 Agent 的责任，生产环境可靠性不够

## 工具设计的具体教训（来自 Anthropic SWE-bench 实践）

**反直觉结论：花在工具设计上的时间比花在 Prompt 上的时间还多。** LLM 调用工具的本质是"读描述 → 决定怎么用"，工具描述和参数设计对 Agent 表现的影响比 Prompt 更大。

两个具体教训：

1. **相对路径 vs 绝对路径**：Agent 移动目录后会搞混相对路径，改成只接受绝对路径后错误消失
2. **写 diff 比重写整个文件更容易出错**：diff 需要精确计算行号，LLM 经常算错；让 LLM 在 JSON 里写代码比在 Markdown 里写代码也更容易出错（需要转义引号和换行）

三条工具设计原则：
- 格式要接近 LLM 在网上见过的自然格式
- 参数设计要防呆（让错误用法变得不可能）
- 像给初级开发写文档一样写工具描述（包括示例、边界情况）

## 必须用 Agent 而不能用 Workflow 的真实场景

**场景：修复 GitHub Issue**

**为什么 Workflow 搞不定？**
- 不知道要改几个文件——可能是 1 个，可能是 5 个
- 不知道路径——先看哪里、再查什么，取决于每一步的发现
- 中间结果不可预测——读了一个文件后发现不是它的问题，要换方向
- 每一步的行动类型都不确定——可能要读代码、搜索关键词、运行测试、查看日志

**判断口诀：** 如果你能提前画出完整的流程图，用 Workflow；如果你画不出来（因为路径取决于中间结果），就用 Agent。

**补充理解：** 外层流程可以固定成 Workflow（搜索 → 修改 → 检查 → 提交），但每一步内部的执行路径完全不可预测，需要 Agent 自主决策。所以本质是"Workflow 外壳 + Agent 内核"。**关键不是外层框架长什么样，而是执行过程中 LLM 是否需要持续做不可预测的决策。**

## RAG 分块策略选择原则

**固定大小 + Overlap 是最常用的通用 Baseline 方案。** 学术研究发现固定大小分块在很多基准测试中表现不亚于语义分块，不需要一味追求更复杂的策略。

**核心原则：固定大小 + Overlap 是基线，围绕数据的天然语义单元来设计分块。** 比如有明确章节结构的文档用结构感知分块，短文本用句子分块。

### 语义分块 vs LLM 驱动分块

两者都跟"语义"有关，但实现方式完全不同：

- **语义分块**：用 Embedding 模型计算相邻句子的向量相似度，相似度低于阈值就切一刀。是数学计算，成本低，但只能看相邻句子的局部相似度
- **LLM 驱动分块**：直接让 LLM 读文本决定在哪里切。能理解全文语境、逻辑结构、主题转换，判断力最强，但成本高、延迟大

一句话总结：语义分块是用数值（向量距离）判断，LLM 驱动分块是用语言理解能力判断。

## LangChain RAG 流水线五步

**Load → Split → Store → Retrieve → Generate**（加载 → 分割 → 存储 → 检索 → 生成）

**注意：** Reranking 不是独立步骤，是 Retrieve 的可选增强（粗筛 + 精排）。

**记忆方法：** 和 RAG 四阶段的映射关系——
- Load + Split + Store = Ingestion（索引阶段）
- Retrieve = Retrieval（检索阶段）
- Generate = Augmentation + Generation（增强 + 生成阶段）

## 混合搜索与 Reranking

### 纯向量检索（语义搜索）的局限

- 对专有名词、缩写、产品名称、错误码等精确匹配场景检索不准
- 比如 user 搜 "HTTP 404"，语义搜索可能返回"常见网络错误"的文档，而不是精确包含 "404" 的那段
- 所以需要混合搜索：语义搜索捕捉"意思接近"，关键词搜索捕捉"精确匹配"

### Reranking 的两种模型架构

**不是 Dense Search 和 Sparse Search**（那是两种检索方式），而是：

- **Bi-encoder（双塔模型）**：query 和 doc 各自独立编码成向量，算余弦相似度。速度快但精度有限（两个文本不互相看）
- **Cross-encoder（交叉编码器）**：query 和 doc 拼在一起同时编码。精度高（能看两者的交互），但速度慢

**为什么不能直接用 Cross-encoder？** 因为要把 query 和每个候选文档拼一起过一遍模型，100 万条文档就要跑 100 万次，太慢。所以先用 Bi-encoder 快速筛出 top-100，再用 Cross-encoder 对这 100 个精排，选出最终 top-10。

---

## RAG / 微调 / 长上下文 — 解决的是不同维度的问题

- **RAG** 解决的是**知识**问题——模型知道什么（适合企业内部知识库）
- **微调** 解决的是**行为**问题——模型怎么输出（适合固定格式输出，如企业文档生成）
- **长上下文** 解决的是**全局**问题——能不能看到完整信息（适合对单篇长文档的全局理解）

三者天然可以组合。经典例子：用 RAG 提供企业知识 + 用微调控制输出格式，生成企业内部报告。

---

## RAG 生成偏离排查

检索到了正确内容但生成偏离，从检索端和生成端两个方向排查：

**检索端：**
- top-k 值设太大，混入不相关噪声分块干扰 LLM

**生成端：**
- Prompt 约束不够强（加"不知道就说不知道"减少幻觉）
- **模型先验干扰**：LLM 自身训练数据里有相关知识，会"自信"地用自己的知识覆盖检索内容
- **Few-shot 示例不当**：示例的风格或内容引导 LLM 偏离

**排查三因口诀：Prompt 不够强、top-k 有噪声、模型先验干扰。**

## Embedding 模型选择维度

选 Embedding 模型就是挑"翻译器"（把文本变成向量），看四件事：

1. **向量维度**：越长越精细但越贵越慢。看预算和数据量选
2. **多语言支持**：中文知识库要选支持中文的（如 BGE 系列），OpenAI 的也行但成本更高
3. **领域特化**：医疗、法律等特殊领域有专门训练过的模型，效果更好
4. **跑分（MTEB 排行榜）**：选模型前去 MTEB 看看，同价位里挑分高的

常见模型：OpenAI text-embedding（通用）、BGE 系列（开源，中文友好）、E5 系列（微软，多语言）

## 微服务架构

### 幂等性的准确定义

不是"同一个操作只会被执行一次"，而是**"同一个操作执行一次和执行多次，效果一样"**。区别：操作可能因重试被执行多次，但不管执行几次，结果跟只执行一次一样。不是说"保证只执行一次"，而是"执行多次也没关系"。

### 熔断器状态名称容易搞反

**关闭=正常通行**（像电路开关，关上=电路通），**打开=断开**（保险丝跳了）。直觉容易反过来。

### 雪崩效应

超时机制的问题不仅仅是"线程池被占满"，术语叫**雪崩效应**：B 挂了，A 的线程全卡在等 B 超时，A 也挂了，调用 A 的 C 也挂了。熔断器的价值是**快速失败**——不等超时，直接返回错误，保护调用方不被拖死。

### Saga 模式的合适例子

旅行预订的例子不够现实（不会因为租不到车就取消机票酒店）。更自然的例子是**跨行转账**：A 银行扣款 → B 银行入账失败（账户冻结）→ 补偿：A 银行加回金额。

### 分布式链路追踪补充

- Trace ID 通过 HTTP 请求头透传（如 `X-Trace-Id`），框架层自动处理
- 常用工具：Jaeger、Zipkin、**SkyWalking**（国内用得最多，Apache 项目）

---

### 服务拆分的两种主流思路

| 俗称 | 正式名称 | 说明 |
|------|---------|------|
| 按不同主题/模块拆 | **按业务能力拆**（Martin Fowler 推荐） | 看公司"做什么事"，按功能模块拆。如订单、支付、库存各自一个服务 |
| 按领域边界拆 | **按子域拆**（DDD Bounded Context） | 看业务领域"边界在哪里"，同一个概念在不同上下文里有不同含义，按含义边界拆 |

**一句话区别：** 按业务能力是"按功能模块拆"，按子域是"按概念边界拆"。实际结果经常差不多，但 DDD 能发现更隐晦的拆分点。

**按子域的例子：** "商品"在目录上下文里有描述和图片，在库存上下文里有数量和仓库位置，在定价上下文里有价格和折扣。三个上下文各有各的模型，各自成为独立服务。

## Prompt Engineering

### ICL 的标准定义（反复遗漏）

不是说"加上下文、提质量"（太笼统），而是：**不更新模型任何参数，仅通过在 prompt 中提供示例，让模型学会执行新任务。**

三个要素缺一不可：**不改权重 + 提供示例 + 学会新任务**

### Few-Shot 核心价值（被纠正 3 次，重点记忆）

**不是"教模型新知识/学会新任务"，而是"消除歧义"。**

模型本来就会情感分析，Few-Shot 不是教会它新技能。问题是你说"分析情感"，模型不知道你要 positive/negative 还是一段中文分析还是 1-5 星评分。示例给了之后格式明确了。

口诀：**Few-Shot 不教新知识，只说"我要什么格式"。本质是激活预训练中已有的模式，帮你"点菜"而不是"教厨师做菜"。**

---

### System Prompt vs User Prompt 区别（功能维度容易漏）

**功能维度：**

| 维度 | System Prompt | User Prompt |
|------|--------------|-------------|
| 内容 | 角色设定、输出格式、安全规则 | 具体问题、输入数据、临时指令 |
| 生命周期 | 持久，贯穿整个对话 | 每轮可变 |
| 类比 | 公司制度（不能改，必须遵守） | 每天的具体任务 |

**底层机制（不是笼统的"预训练见过"）：**

1. **特殊 Token 标记**：API 插入特殊 token（如 `<|im_start|>system`），模型通过标记区分角色
2. **注意力权重差异**：模型给 system prompt 的 token 分配更高的注意力权重，所以 system 指令对输出影响更大
3. 训练数据中 system-like 指令和用户查询有明确分隔，模型学会了区分"全局规则"和"当前任务"

---

### CoT 提升推理能力的三个原因（只记到 1.5 个）

口诀：**拆步骤 + 多计算 + 能纠错**

1. **逐步计算替代一次性计算**：没有 CoT 时模型要从输入直接跳到答案，超出单步能力。CoT 把复杂推理拆成多个小步骤
2. **生成中间步骤 = 更多的计算层**：每生成一个 token = 一次前向传播。CoT 多生成 N 个 token = 多做 N 次前向传播 = 多了 N 轮"思考"。本质是用更多计算换更高准确率
3. **自回归的纠错能力**：每个中间步骤都成为后续步骤的上下文，某步有误时后续步骤可能通过注意力机制纠正

### CoT 的三个局限性（完全不知道）

1. **推理成本上升**：增加输出 token 数，费用和延迟都增加
2. **简单任务反而出错**：过度推理，把本来能直接答对的问题想复杂了
3. **推理过程不一定是"真实思考"**：只是看起来像推理的文本，不一定是模型真正的内部推理过程

### Constrained Decoding 的 Schema 传入方式

不是放在提示词里（那是第 1 代做法），而是通过**单独的 API 参数**传入。如 OpenAI 的 `response_format` 参数。API 内部把 Schema 编译成有限状态机，在解码时逐 token 强制约束。

### JSON Mode 的原理（不知道）

JSON Mode 只约束输出必须是合法 JSON 语法（`{}` 配对、引号配对），但**不知道你的 Schema 长什么样**。所以可能出现字段名不对、类型不对。对比：JSON Mode 保证语法合法，Constrained Decoding 保证结构符合 Schema。

### Fine-Tuning 的适用场景（说得太窄）

不只是"固定格式"。三个典型场景：
- **改变模型行为/风格**：让模型用特定的语气回答
- **领域适配**：用医疗数据微调，让模型在医疗场景表现更好
- **降低推理成本**：小模型微调后可以匹配大模型效果，省 token 费

## 锚点

想象你雇了一个维修工来修房子。他不会站在门口念一份维修报告。他会走进去，看到墙裂了，拿锤子敲敲，发现是水管问题，去拿扳手，修完检查一下，再去看下一个地方。

Agent 就是这个维修工。LLM 是他的脑子，工具（Tools）是他的工具箱，工作流（Workflow）是你提前画好的路线图，而 Agent 模式是他自己判断先去哪、用什么工具。

整条线是这样的：LLM 本身只会说话 → 给它装上工具，它能干活了 → 任务简单就按固定路线走（Workflow）→ 任务复杂就让它自己判断怎么走（Agent）→ 任务大到一个人干不了就叫一队人（Multi-Agent）。

## 核心问题

LLM 能写文章、能聊天，但它只能输出文字。现实任务需要行动：查数据库、调 API、读文件、发邮件。怎么让一个只会"说话"的东西变成能"做事"的？

答案是分两步：第一步，给它装工具（Tool Use）；第二步，给它一个决策框架，让它知道什么时候该用什么工具、按什么顺序用。

这个决策框架有两种风格：一种是你提前定好的（Workflow），一种是 LLM 自己决定的（Agent）。

## 核心概念

### Augmented LLM：加了装备的 LLM

一个普通的 LLM 只能做一件事：输入文本，输出文本。

但如果你给它装上三样东西——检索能力（Retrieval）、工具调用（Tools）、记忆（Memory）——它就变成了一个 Augmented LLM。这是所有 Agent 系统的基本单元，不管多复杂的架构，底层都是这个。

```
普通 LLM:  用户输入 → LLM → 文本输出
增强 LLM:  用户输入 → LLM → 调用工具/检索记忆 → 获取结果 → LLM → 最终输出
```

这个概念为什么重要？因为后面所有的 Workflow 和 Agent 模式，都是在这个增强 LLM 基础上搭出来的。如果你不理解这个基础，后面的编排模式就成了空中楼阁。

既然增强 LLM 能做这么多事，那多个增强 LLM 怎么协作？这引出了接下来的两种编排风格。

### Workflow vs Agent：两条路线

Anthropic 做了一个很关键的区分：

- **Workflow**：代码提前定义好了路径。LLM 在这条路径上的某个节点干活，但走哪条路是代码说了算。
- **Agent**：LLM 自己决定下一步做什么。代码只提供工具和边界条件，决策权在 LLM 手里。

打个比方：Workflow 是地铁，路线固定，你在哪站下车是提前定好的；Agent 是打车，你告诉司机目的地，他决定走哪条路、怎么绕开堵车。

什么时候用哪个？原则很简单：**能不用就不用**。很多时候优化一个单次 LLM 调用就够了。如果任务确实需要多步，先看能不能用 Workflow（可控、可预测），只有当任务太灵活、路径无法预测时才上 Agent。

## Workflow 的五种模式

Workflow 的好处是可控。路径固定，每一步做什么都知道，调试方便，成本和延迟也可预期。

### 1. Prompt Chaining（提示链）

把一个大任务拆成一串小任务，前一个的输出是后一个的输入。中间可以加程序检查（Gate），确认上一步没问题才继续。

```
输入 → LLM-1 → [检查] → LLM-2 → [检查] → LLM-3 → 输出
```

**例子**：先让 LLM 生成营销文案，检查没问题，再让它翻译成日语。

**适合**：任务能干净地拆成固定步骤。每一步变小了，准确率更高，但牺牲了速度（串行执行）。

### 2. Routing（路由）

先分类，再分发。不同类型的输入走不同的处理流程。

```
输入 → 分类器 → 路径A（专门处理A类问题）
              → 路径B（专门处理B类问题）
              → 路径C（专门处理C类问题）
```

**例子**：客服系统——退款问题走退款流程，技术问题走技术支持流程，一般问答直接回答。也可以用来做模型路由：简单问题用 Haiku（便宜快），复杂问题用 Sonnet（贵但强）。

**适合**：输入有明确的类别区分，不同类别需要不同的处理逻辑。不路由的话，一个 Prompt 很难同时擅长所有类别。

### 3. Parallelization（并行）

两种玩法：

- **Sectioning**：把任务拆成独立子任务，同时跑，结果合并。
- **Voting**：同一个任务跑多次，取多数意见。

```
Sectioning:  输入 → [LLM-A 处理子任务1] ─┐
                   [LLM-B 处理子任务2] ─┼→ 合并 → 输出
                   [LLM-C 处理子任务3] ─┘

Voting:  输入 → [LLM-1 判断] ─┐
               [LLM-2 判断] ─┼→ 多数投票 → 输出
               [LLM-3 判断] ─┘
```

**例子**：代码审查——一个 LLM 检查安全问题，一个检查性能问题，一个检查可读性，最后汇总。Voting 的例子：内容审核时多个 LLM 分别判断是否违规，超过半数才判定违规（降低误判）。

**适合**：子任务之间没依赖关系（提速），或者需要多角度交叉验证（提准）。

### 4. Orchestrator-Worker（编排者-工人）

一个"包工头" LLM 动态拆任务，分给多个"工人" LLM 去做，最后汇总结果。

```
输入 → 编排者 LLM → [工人A] ─┐
                   → [工人B] ─┼→ 汇总 → 输出
                   → [工人C] ─┘
```

这跟 Parallelization 的区别在于：Parallelization 的子任务是提前定义好的，Orchestrator-Worker 的子任务是 LLM 根据具体输入动态决定的。

**例子**：代码修改——编排者分析需求后决定要改哪几个文件、每个文件怎么改，工人分别去改，最后汇总。

**适合**：你没法提前预测需要多少子任务、每个子任务是什么。比如改代码，改几个文件取决于任务内容。

在 LangGraph 中，这用 `Send` API 实现：编排者返回一个 `Send` 列表，每个 `Send` 创建一个工人节点，工人的输出写入共享状态。

### 5. Evaluator-Optimizer（评估者-优化者）

一个 LLM 生成结果，另一个 LLM 评估并给反馈，不通过就重新生成，循环直到满意。

```
输入 → 生成者 LLM → 评估者 LLM → [通过] → 输出
                     ↑    ↓ [不通过，给反馈]
                     └────┘
```

**例子**：文学翻译——翻译者 LLM 出第一版，评估者指出"这句的语气不对"，翻译者修改后再评。

**适合**：有明确的评估标准，而且多改几轮真的能变好。判断标准：如果你作为人类能给反馈让它变好，那另一个 LLM 大概率也能。

### 五种 Workflow 速查

| 模式 | 核心思路 | 路径 | 适合场景 |
|------|---------|------|---------|
| Prompt Chaining | 串行分步 | 固定 | 能拆成固定步骤的任务 |
| Routing | 先分类再处理 | 固定分支 | 输入有明确类别 |
| Parallelization | 同时干多件事 | 固定并行 | 子任务独立或需交叉验证 |
| Orchestrator-Worker | 动态拆任务 | 动态子任务 | 子任务数量和内容不确定 |
| Evaluator-Optimizer | 做了检查重做 | 循环 | 有明确评估标准、迭代有效 |

## Agent 模式

### ReAct：边想边做

ReAct（Reasoning + Acting）是 Agent 最经典的模式，来自 Yao et al. 2023 的论文。

它的核心思路很简单：LLM 交替进行"思考"和"行动"。

```
Thought: 用户问的是北京天气，我需要查询天气 API
Action:  调用天气 API，参数 city="北京"
Observation: 北京今天 25°C，晴
Thought: 已经拿到天气信息，可以回答了
Answer:  北京今天 25°C，晴天
```

为什么 Thinking 步骤有用？实验证明，去掉 Thought 只保留 Action，准确率明显下降。因为在生成行动之前先推理一遍，能减少"盲目行动"。

ReAct 的循环是：**Think → Act → Observe → Think → Act → ... → 回答**。每一步都有环境反馈（Observation），不是闭眼瞎猜。

**适合**：需要跟外部环境交互的任务（搜索、查 API、读文件），步骤不多但需要根据中间结果调整策略。

### Plan-and-Execute：先规划再执行

ReAct 的问题是"走一步看一步"，缺乏全局视野。碰到复杂任务容易走偏。

Plan-and-Execute 把"规划"和"执行"拆成两个独立阶段：

```
输入 → 规划者 LLM → [步骤1, 步骤2, 步骤3, ...]
                        ↓
       执行者 LLM ← 逐步执行每个步骤
                        ↓
       [可选] 重新规划（如果某步失败了）
```

跟 ReAct 的区别：

| 维度 | ReAct | Plan-and-Execute |
|------|-------|-------------------|
| 决策方式 | 走一步看一步 | 先想好全部步骤再动手 |
| 全局视野 | 弱，只看当前步 | 强，有完整计划 |
| 灵活性 | 高，随时调整 | 中，按计划走，出错才重新规划 |
| 适合 | 简单、步骤少的任务 | 复杂、步骤多的任务 |

Plan-and-Execute 还支持"重新规划"：如果执行到第 3 步发现前提错了，可以回去让规划者重新出方案。

### Multi-Agent：多人协作

当任务大到单个人搞不定时，就用多个 Agent 分工合作。常见的协作模式有三种：

**Supervisor（主管模式）**

一个中心 Agent 负责接收任务、分配给专业 Agent、汇总结果。

```
用户 → 主管 Agent → [搜索 Agent] ─┐
                  → [编码 Agent] ─┼→ 主管汇总 → 返回用户
                  → [分析 Agent] ─┘
```

**Hierarchical（层级模式）**

主管模式的基础上再加层级：大主管管小主管，小主管管工人。

```
用户 → 大主管 → [小主管A → [工人1, 工人2]]
              → [小主管B → [工人3, 工人4]]
```

**Swarm / Handoff（接力模式）**

没有固定的主管。Agent 之间直接交接任务，像接力赛一样。一个 Agent 觉得"这个该另一个 Agent 处理"，就直接把对话交给它。

```
用户 → Agent A（处理到某个点）→ Handoff → Agent B → Handoff → Agent C → 返回用户
```

三种模式对比：

| 模式 | 控制方式 | 适合场景 | 劣势 |
|------|---------|---------|------|
| Supervisor | 中心化 | 任务类型明确、需要统一汇总 | 主管是瓶颈 |
| Hierarchical | 多层中心化 | 超大规模任务 | 复杂度高 |
| Swarm/Handoff | 去中心化 | 对话式、需灵活流转 | 难追踪、难调试 |

## Function Calling 的底层机制

面试官问 Function Calling，他想听的不是"调用了一个函数"，而是"你知道 LLM 本身并不执行代码"。

整个流程是这样的：

**1. 注册工具**

你定义工具的名字、描述、参数 schema（JSON Schema 格式），告诉 LLM "你有这些工具可用"。

**2. 注入 Prompt**

这些工具定义被注入到 LLM 的上下文中（通常是 System Prompt 或者特殊的 token 段）。LLM 看到的本质是一段结构化文本，描述了每个工具能做什么、需要什么参数。

**3. LLM 决策**

LLM 根据用户输入和工具描述，决定要不要调工具、调哪个、传什么参数。注意：**LLM 只输出一个结构化的"调用意图"**，它不执行任何代码。

输出的格式类似：
```json
{"name": "get_weather", "arguments": {"city": "北京"}}
```

**4. 应用层执行**

你的应用程序解析这个 JSON，调用真正的函数，拿到结果。

**5. 结果回传**

把函数执行结果作为一条特殊消息（tool_result）追加到对话中，LLM 看到结果后生成最终回复。

```
用户输入 → LLM（看到工具列表）
         → 输出：{"name": "get_weather", "args": {"city": "北京"}}
         → 应用层调用天气 API → 结果：25°C 晴
         → LLM（看到工具结果）→ "北京今天 25°C，晴天"
```

**关键认知**：LLM 从头到尾没有执行任何代码。它只是一个决策器——决定调什么、传什么参数。执行是外部应用层的事。这也解释了为什么工具的描述和参数设计这么重要：LLM 全靠读描述来决定怎么用。

模型层面，Function Calling 能力是通过微调训练得到的。训练数据包含了大量"用户提问 → 正确的工具调用意图"的配对，让模型学会什么时候该调工具、什么时候直接回答。

## Planning 技术

Agent 要完成任务，需要先把大目标拆成小步骤。这就是 Planning。

### Chain of Thought（CoT）

让 LLM "一步一步想"。不是直接给答案，而是先写出推理过程。

为什么有效？因为 LLM 是逐 token 生成的。如果直接给答案，它只有一次机会。如果先写推理过程，每一步生成都给了下一步更多上下文，相当于用更多的计算换取更高的准确率。

### Tree of Thoughts（ToT）

CoT 的升级版。不是只走一条推理链，而是在每一步生成多个可能的思路，形成一棵树，用 BFS 或 DFS 搜索最优路径。

```
CoT: 问题 → 思路1 → 思路2 → 思路3 → 答案

ToT: 问题 → [思路1A, 思路1B, 思路1C]
              ↓
         [思路2A, 思路2B]  [思路2C]
              ↓
           答案
```

适合需要探索多条路径的任务，比如创意问题、复杂的数学推理。

### 自我反思（Self-Reflection）

Agent 做完一件事后回头看："我刚才做得对吗？"不对就修正。

代表工作：

- **Reflexion**：Agent 执行完一轮后，如果失败了，生成一段"反思"（哪里做错了、下次怎么改），存进记忆。下次执行时带着反思一起，避免重复犯错。
- **ReAct 的 Thought 步骤**本身就是一种轻量级的自我反思：在行动前先想想为什么要这么做。

## Memory 系统

Agent 需要记住东西，否则每次交互都从零开始。

跟人类记忆类比：

| 人类记忆 | Agent 中的对应 |
|---------|--------------|
| 感觉记忆 | 输入的 embedding 表示 |
| 短期记忆 | 上下文窗口（Context Window）里的内容 |
| 长期记忆 | 外部向量数据库，按需检索 |

短期记忆就是对话历史和当前 Prompt，受限于上下文窗口长度。长期记忆是把信息存进向量数据库（如 Pinecone、FAISS），需要时通过相似度搜索取出来放到上下文中。

检索时的排序考虑三个因素：
- **相关性**（Relevance）：跟当前任务有多相关
- **时效性**（Recency）：越新的信息权重越高
- **重要性**（Importance）：核心记忆 vs 琐碎事件，可以用 LLM 直接打分

## 什么时候用什么：决策框架

面试中常问"怎么选型"，这里给出一个从简到繁的决策路径：

```
1. 单次 LLM 调用能解决？ → 直接用，不用 Agent
   ↓ 不够
2. 任务能拆成固定步骤？ → Prompt Chaining
   ↓ 不够
3. 不同输入需要不同处理？ → Routing
   ↓ 不够
4. 子任务之间独立？ → Parallelization
   ↓ 不够
5. 子任务不确定，需要动态分配？ → Orchestrator-Worker
   ↓ 不够
6. 结果需要反复打磨？ → Evaluator-Optimizer
   ↓ 不够
7. 路径完全无法预测？ → Agent（ReAct 或 Plan-and-Execute）
   ↓ 任务大到单 Agent 搞不定
8. Multi-Agent
```

核心原则（来自 Anthropic）：

1. **保持简单**。能用简单方案就别上复杂架构。
2. **透明**。Agent 的规划过程要可见，方便调试和信任。
3. **工具设计很重要**。ACI（Agent-Computer Interface）值得投入跟 HCI（Human-Computer Interface）一样多的精力。工具描述要清晰、参数要防错（Poka-yoke）、格式要对 LLM 友好。

## 逐节详解

### Section 1: 为什么不能只用裸 LLM

裸 LLM 的问题不是"不够聪明"，而是"没有手脚"。它能分析问题、生成方案，但不能执行。你不能让它"帮我查一下订单状态"，因为它没有查订单的能力。

Augmented LLM 解决的就是这个问题：给它装上工具（API、代码执行器）、记忆（向量库、对话历史）、检索（搜索引擎、RAG），它就从一个只会说话的模型变成了一个能干活的系统。

但装上工具只是第一步。有了工具之后，怎么组织多个 LLM 调用、怎么编排工作流、什么时候让 LLM 自己决策——这些才是 Agent 架构要解决的问题。

### Section 2: 从 Workflow 到 Agent 的渐进路径

Anthropic 的观察是：**最成功的 Agent 实现往往用的是最简单的模式**。

复杂度是敌人。每多一层抽象，就多一层调试难度、多一个出错点。所以推荐的思路是：

先确认单次 LLM 调用 + 好的 Prompt + 检索能否解决。很多时候这就够了。

如果不行，看任务结构。如果步骤固定（比如"先写提纲、再写正文、最后润色"），用 Prompt Chaining。如果步骤取决于输入类型（比如客服分流），用 Routing。

只有当任务的路径完全无法预测（比如"帮我修这个 GitHub issue，不知道要改几个文件"），才需要 Agent。Agent 把路径决策交给了 LLM，获得了灵活性，但也失去了确定性。

### Section 3: ReAct 的设计哲学

ReAct 的核心洞察：推理（Reasoning）和行动（Acting）应该交替进行，而不是分开。

如果只有行动没有推理（Act-only），LLM 就像盲人摸象，看到什么就反应什么，没有全局判断。如果只有推理没有行动（Think-only），LLM 就变成了纸上谈兵，想得很美但什么都不做。

ReAct 让两者交替：

```
用户："Python 之父是谁？"
Thought: 这是一个事实性问题，我需要搜索确认
Action:  search("Python creator")
Observation: Guido van Rossum
Thought: 确认了，可以回答
Answer:  Python 的创始人是 Guido van Rossum
```

这个循环可以重复很多次。每一步的 Observation 提供了"真实世界"的反馈，避免 LLM 闭门造车。

面试时要注意区分：ReAct 是一种 Agent 模式（LLM 自主决策循环），不是 Workflow（路径不固定，由 LLM 动态决定）。

### Section 4: 工具设计为什么比 Prompt 更重要

Anthropic 在构建 SWE-bench Agent 时发现：**花在工具设计上的时间比花在 Prompt 上的时间还多**。

为什么？因为 LLM 调用工具的本质是"读描述 → 决定怎么用"。如果描述不清楚，或者参数设计容易出错，LLM 就会用错。

具体教训：
- 相对路径 vs 绝对路径：LLM 在 Agent 移动目录后会搞混相对路径。改成只接受绝对路径后，错误消失。
- 格式开销：让 LLM 写 diff（需要精确计算行号）比让 LLM 重写整个文件更容易出错。让 LLM 在 JSON 里写代码（需要转义引号和换行）比在 Markdown 里写代码更容易出错。

工具设计原则：
- 给 LLM 足够的 token 去"想"，别让它写到一半发现写错了但改不了
- 格式要接近 LLM 在网上见过的自然格式
- 参数设计要防呆（Poka-yoke），让错误用法变得不可能
- 像给初级开发写文档一样写工具描述：包括示例、边界情况、输入格式要求

### Section 5: Multi-Agent 的实际取舍

Multi-Agent 听起来很酷，但它是最后手段，不是首选。

为什么？因为每多一个 Agent，系统复杂度就多一层：
- Agent 之间的通信成本（Prompt 传递、结果序列化）
- 调试困难（哪个 Agent 出的错？）
- 成本增加（每个 Agent 都是独立的 LLM 调用）
- 一致性问题（多个 Agent 可能对同一件事有不同理解）

实际中，Supervisor 模式最常用：一个 Agent 负责理解和分发，其他 Agent 各管一摊。LangGraph 的 `Send` API 就是实现 Supervisor 模式的标准方式。

Swarm/Handoff 模式（如 OpenAI 的 Swarm 框架）更轻量，适合对话式场景，但在生产环境中可靠性不如 Supervisor。

## 与其他方法的对比

### Workflow vs Agent：它们要解决的是同一个问题吗？

不完全一样。

Workflow 解决的是"怎么把一个复杂任务编排好"，路径是预设的，LLM 在固定节点上干活。Agent 解决的是"怎么让 LLM 自主完成任务"，路径是动态的，LLM 自己决定下一步。

但它们有重叠：Orchestrator-Worker Workflow 看起来像 Agent（编排者动态决策），但它的决策范围有限（只决定拆分方式，不决定后续流程）。真正的 Agent 是在整个过程中持续决策的。

### ReAct vs Plan-and-Execute

它们要解决的是同一个问题：怎么让 LLM 完成多步骤任务。区别在于规划时机。

ReAct 是"边走边看"，每一步都重新思考下一步该做什么。好处是灵活，能根据中间结果随时调整。坏处是没有全局视野，可能在错误的方向上越走越远。

Plan-and-Execute 是"先想好再走"，一开始就制定完整计划，然后按计划执行。好处是方向明确，不会跑偏。坏处是如果初始理解有误，整个计划可能白费。

实际中常混合使用：Plan-and-Execute 做大框架，每步执行时用 ReAct 的 Think-Act-Observe 循环。

### 不同规模的选型对比

| 任务特征 | 推荐方案 | 原因 |
|---------|---------|------|
| 单一问题，有明确答案 | 单次 LLM 调用 | 最简单，够用 |
| 多步骤，步骤固定 | Prompt Chaining | 可控、可调试 |
| 不同类型输入 | Routing | 分类后针对性处理 |
| 多步骤，步骤独立 | Parallelization | 并行提速 |
| 多步骤，步骤不确定 | Orchestrator-Worker | 动态拆分 |
| 需要反复打磨质量 | Evaluator-Optimizer | 循环改进 |
| 路径不可预测 | ReAct Agent | 灵活应对 |
| 复杂任务，需要全局规划 | Plan-and-Execute | 有全局视野 |
| 超大规模，单 Agent 搞不定 | Multi-Agent (Supervisor) | 分工协作 |

## 面试高频考点

| 考点 | 面试官真正想知道的 | 回答要点 | 常见错误 |
|------|-------------------|---------|---------|
| ReAct 是什么 | 你是否理解 Agent 的核心循环 | Think-Act-Observe 循环，推理和行动交替，比纯行动准确率高 | 把 ReAct 说成框架而不是模式 |
| Workflow 和 Agent 的区别 | 你是否会做架构选型 | Workflow 路径固定、Agent 路径动态；Workflow 可控、Agent 灵活 | 认为所有多步骤 LLM 调用都是 Agent |
| 什么时候该用 Agent | 你有没有过度设计的倾向 | 路径不可预测时才用；能简单就简单 | 什么任务都想上 Agent |
| Function Calling 的原理 | 你是否知道 LLM 不执行代码 | LLM 只输出调用意图（JSON），应用层执行，结果回传 LLM | 以为 LLM 直接执行函数 |
| Multi-Agent 的模式 | 你是否了解协作架构的取舍 | Supervisor 最常用；Swarm 轻量但难调试 | 只说概念不讲取舍 |
| Prompt Chaining vs Parallelization | 你能不能区分串行和并行的适用场景 | 串行：步骤有依赖；并行：步骤独立 | 混淆两者适用条件 |
| Orchestrator-Worker 和 Parallelization 的区别 | 你理解动态 vs 静态拆分 | 子任务是否可预测：Parallelization 可预测，Orchestrator 不能 | 认为两者一样 |
| Agent 的工具设计 | 你有没有实际踩过坑 | 工具描述要像给初级开发写文档；防呆设计；格式要对 LLM 友好 | 只关注 Prompt 不关注工具 |
| Plan-and-Execute 的优势 | 你理解全局规划的价值 | 先出完整计划再执行，支持重新规划；比 ReAct 有全局视野 | 跟 ReAct 完全等同 |
| Agent 的 Memory 怎么做 | 你理解上下文限制的应对方式 | 短期记忆=上下文窗口，长期记忆=向量库+检索；检索按相关性/时效/重要性排序 | 只说"用向量数据库"不解释为什么 |

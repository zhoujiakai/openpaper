### Q1 [选择题] RAG 的完整链路包含哪四个阶段？

A. 加载 → 分割 → 存储 → 生成
B. Ingestion → Retrieval → Augmentation → Generation
C. 查询 → 嵌入 → 搜索 → 回答
D. 分块 → 向量化 → 相似度计算 → 输出

- **正确答案**：B
- **评分要点**：选择了正确选项即通过
- **知识点**：RAG 四阶段链路

### Q4 [简答题] 为什么 RAG 系统中，入库和检索必须使用同一个 Embedding 模型？

- **参考答案**：不同 Embedding 模型产生的向量在不同的语义空间中，维度和含义都可能不同。用 A 模型入库、B 模型检索，相当于在两个不兼容的坐标系中比较距离，相似度计算结果没有意义。
- **评分要点**：
  1. 必须提到"不同的向量空间/语义空间"
  2. 需要说明跨空间比较没有意义
  3. 加分项：提到维度可能也不同
- **知识点**：Embedding 模型一致性

### Q5 [简答题] 什么是 Reranking？为什么检索后还需要重排？

- **参考答案**：Reranking 是两阶段检索的第二阶段。第一阶段用双塔模型快速粗筛（速度快但精度一般），第二阶段用 cross-encoder 对候选进行精确打分排序（精度高但速度慢）。需要 Reranking 是因为粗筛用的双塔模型对 query 和 doc 独立编码，精度有限；cross-encoder 将 query 和 doc 拼接后联合编码，能捕捉更细粒度的相关性。
- **评分要点**：
  1. 必须提到"两阶段检索"概念
  2. 需要解释双塔模型 vs cross-encoder 的区别（独立编码 vs 联合编码）
  3. 加分项：提到速度和精度的权衡
- **知识点**：Reranking 与两阶段检索

### Q6 [简答题] 简述 RAG 相比微调（Fine-tuning）的三个核心优势。

- **参考答案**：
  1. 知识可即时更新（更新索引即可，无需重训模型）
  2. 成本更低（只需维护索引和检索，不需要 GPU 训练）
  3. 可解释性更强（可以展示回答基于哪些文档，支持溯源审计）
- **评分要点**：
  1. 至少提到两个优势
  2. 每个优势需要说清"为什么 RAG 有而微调没有"
  3. 加分项：提到 RAG 和微调可以组合使用
- **知识点**：RAG vs 微调

### Q7 [对比题] 比较 dense search（纯向量语义搜索）和 sparse search（关键词搜索如 BM25），各适合什么场景？

- **参考答案**：
  | 维度 | Dense Search | Sparse Search（BM25） |
  |------|-------------|---------------------|
  | 原理 | 用向量相似度衡量语义相关性 | 用词频-逆文档频率衡量关键词匹配 |
  | 擅长 | 同义改写、语义理解 | 精确匹配专有名词、错误码、产品型号 |
  | 弱点 | 对精确标识符不敏感 | 不懂同义词、语义相近但用词不同 |
  | 适用场景 | 开放式问答、模糊查询 | 搜索特定编号、名称、代码 |
- **评分要点**：至少指出 3 个区别维度（原理、擅长、弱点、场景）
- **知识点**：混合检索原理

### Q8 [应用题] 你在做一个客服知识库 RAG 系统，用户反馈"回答经常不准确"。你发现检索到的文档块其实是对的，但生成的内容却偏离了文档内容。请分析可能的原因并提出优化方案。

- **参考答案**：
  可能原因：
  1. Prompt 没有足够强调"必须基于 CONTEXT 回答"，模型自由发挥
  2. 检索到的文档块太多，噪声干扰了模型判断
  3. 模型自身参数知识干扰（模型更"相信"自己训练时学的知识）

  优化方案：
  1. 加强 Prompt 约束："只使用 CONTEXT 中的信息回答，不要使用你的先验知识"
  2. 减少 top-k 数量，或引入 Reranking 提高检索精度
  3. 在 Prompt 中加入 few-shot 示例，教模型如何基于上下文回答
- **评分要点**：
  1. 正确分析至少 2 个可能原因
  2. 每个原因对应至少 1 个优化方案
  3. 加分项：提到 Prompt 工程是关键杠杆
- **知识点**：RAG 端到端调优

### Q9 [应用题] 公司要做"内部政策问答系统"，文档包括员工手册（结构化 Markdown）、HR 邮件（纯文本）、公司 wiki（HTML）。请设计完整的 RAG 方案，包括分块策略、检索方式、以及为什么这样设计。

- **参考答案**：
  1. 员工手册（Markdown）：用递归结构感知分块，按章节/段落层级切割，保留结构信息
  2. HR 邮件（纯文本）：按邮件主题/对话轮次分块，一封邮件或一轮对话为一个 chunk
  3. 公司 wiki（HTML）：按页面 + 段落分块，HTML 标签提供天然结构

  检索方式：混合搜索（dense + sparse）+ Reranking
  - dense：处理"年假怎么休""请假流程"等语义查询
  - sparse：匹配政策编号、部门名称等精确标识
  - Reranking：提高最终结果精度

  其他设计：
  - 元数据过滤：添加部门、日期、政策类型等元数据，支持筛选
  - Prompt：加入"如果多份政策有冲突，以最新日期为准"的指令
- **评分要点**：
  1. 为每种文档类型选择了合理的分块策略并说明理由
  2. 检索方式覆盖了混合搜索思路
  3. 加分项：提到元数据过滤或 Prompt 设计
- **知识点**：RAG 端到端设计

### Q10 [简答题] Agentic RAG 和传统 RAG 最本质的区别是什么？Agentic RAG 能做哪些传统 RAG 做不到的事？

- **参考答案**：最本质的区别是 Agentic RAG 有**反馈循环**。传统 RAG 是固定流水线：查一次就完。Agentic RAG 让 Agent 编排整个流程，能做的事包括：
  1. Query Rewriting：自动改写模糊或不精确的查询
  2. 工具选择：根据问题类型选择不同检索工具
  3. 结果评估：判断检索结果是否相关，不相关则换策略重试
  4. 信息验证：对检索内容做事实核查
  5. 迭代优化：多轮检索逐步逼近答案
- **评分要点**：
  1. 必须提到"反馈循环"或"迭代优化"的核心概念
  2. 至少列出 3 个 Agentic RAG 的能力
  3. 加分项：提到 GraphRAG 或具体应用场景
- **知识点**：Agentic RAG

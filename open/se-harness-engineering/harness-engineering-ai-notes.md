## 锚点

**Harness Engineering 就是给 AI 搭一个"温室"。**

普通种地靠天吃饭（手写代码），Vibe Coding 是撒完种子就走（写个 prompt 让 AI 写代码，差不多就行）。Harness Engineering 不一样——你建温室、装灌溉系统、设温度湿度传感器，然后种子（Agent）在里面按你的规矩长。

你的主要工作不是种地（写代码），而是**建温室**（设计环境、定约束、搭反馈循环）。代码是 Agent 长出来的，你的职责是让这个温室靠谱。

## 核心问题

**为什么有了 Vibe Coding 和 Prompt Engineering，还需要 Harness Engineering？**

因为 Agent 写代码的速度远超人类审查的速度。Vibe Coding 是"信任 AI，差不多就行"，但当你用 Agent 在 5 个月内生成百万行代码、每天合并 3.5 个 PR 的时候，"差不多"就不够了——你需要一个系统，让 Agent 写出的代码天然就是对的。

OpenAI 的团队用了一个极端的约束来逼自己想清楚这个问题：**0 行手写代码**。产品从第一个 commit 到百万行代码，全部由 Codex 生成。3 个工程师驱动，5 个月完成。

## 核心概念

### 1. 从写代码到搭环境

以前工程师的核心产出是代码。Harness Engineering 里，核心产出是**让 Agent 能高效工作的环境**。

OpenAI 团队发现，早期 Agent 写代码慢，不是 Agent 不行，是**环境没搭好**——缺工具、缺抽象、缺内部结构。工程师的工作变成了：发现 Agent 在哪卡住 → 补上缺失的能力 → 让 Agent 继续干。

具体来说：Agent 写不出某个功能？不是"再试试"，而是问"它缺什么能力？"——是缺少某个脚本？缺少某个 API？缺少某种文档？把缺失的东西补上，Agent 就能自己搞定。

### 2. 上下文管理：给地图，不给说明书

这是 OpenAI 学到的最早的教训之一。

**AGENTS.md 不能写成百科全书。** 原因有四个：
- 上下文窗口是稀缺资源。巨大的指令文件会挤掉任务本身、代码和文档，Agent 反而搞不清重点
- 所有东西都"重要"等于没有重点。Agent 会变成局部模式匹配，而不是有目的地导航
- 指南会过期。一个巨大的手册变成过期规则的坟墓
- 难以验证。单一大块文件没法做自动化检查

**解决方案：AGENTS.md 当目录，docs/ 当百科。**

AGENTS.md 大约 100 行，只放指针和关键约束。详细文档放在结构化的 `docs/` 目录里，Agent 需要时按图索骥。这叫 **Progressive Disclosure（渐进式披露）**——先给小的稳定入口，教 Agent 在哪找更多信息，而不是一开始就塞满。

配套机制：
- 专门的 linter 和 CI job 验证知识库是否最新、交叉链接是否正确
- 定期跑"文档园艺"Agent，扫描过期文档并自动开 PR 修复

### 3. 架构约束：先上紧箍咒，再放权

Agent 在严格边界和可预测结构下表现最好。OpenAI 的做法：

每个业务领域分成固定的层：**Types → Config → Repo → Service → Runtime → UI**，代码只能"向前"依赖。跨领域的东西（认证、连接器、遥测、feature flag）只能通过 Providers 接口进入。

这种架构通常几百人的团队才会建。但在 Agent 写代码的场景里，它是**早期必须品**——约束是让 Agent 快而不乱的前提。

具体执行：
- 自定义 linter（当然也是 Codex 写的）+ 结构化测试
- Lint 错误信息里直接嵌入修复指引（给 Agent 看的）
- 边界严格管控，边界内给 Agent 充分自由
- 不要求代码符合人类审美，只要正确、可维护、对未来 Agent 可读就行

### 4. 反馈循环：Agent 的眼睛和手

Agent 要做有用的活，得能看到结果。OpenAI 做了几件事让 Agent"看得见"：

- **UI 可见**：App 能按 git worktree 启动独立实例，Agent 每个改动有自己的运行环境。接入 Chrome DevTools Protocol，Agent 可以截屏、看 DOM 快照、导航页面——复现 bug、验证修复都能自己来
- **日志可见**：每个 worktree 有独立的临时可观测性栈。Agent 可以用 LogQL 查日志、用 PromQL 查指标。Prompt 可以写"确保服务启动在 800ms 内完成"，Agent 能自己验证
- **Agent-to-Agent 审查**：大部分代码审查由 Agent 之间完成，人类审查不是必须的

这些能力让单个 Codex 运行可以在一个任务上连续工作 6 小时以上（经常是人类睡觉的时候）。

### 5. 漂移治理：像垃圾回收一样清理技术债

Agent 会复制仓库里已有的模式——哪怕这些模式不太好。时间一长，必然漂移。

OpenAI 团队最初每周五花 20% 的时间手动清理"AI slop"（AI 产出的垃圾代码）。 unsurprisingly，这不可持续。

解决方案：**编码"黄金原则" + 定期自动清理**。

黄金原则是具体的、可机械检查的规则。比如：
- 用共享工具包，不要手搓 helper
- 不"YOLO式"探测数据，必须在边界验证或用类型化 SDK

定期跑一组后台 Codex 任务，扫描偏差、更新质量评分、开重构 PR。大多数 PR 一分钟内审完自动合并。

这像**垃圾回收**：技术债像高息贷款，持续小额还清比攒一大笔再还好得多。人类的品味被捕获一次，然后持续执行在每一行代码上。

### 6. 开发者角色的重新定义

在 Harness Engineering 里，人类工程师做的事变成了：
- **确定优先级**：哪些功能先做
- **翻译反馈**：把用户反馈转化为验收标准
- **验证结果**：确认 Agent 做的东西是对的
- **诊断障碍**：Agent 卡住时，找出缺什么（工具、护栏、文档），反馈回仓库

人类始终在循环中，但工作的抽象层级变了——不再是"写代码"，而是"设计系统让 Agent 写出对的代码"。

一个完整的 Agent 工作流：验证代码状态 → 复现 bug → 录视频展示问题 → 实现修复 → 验证修复 → 录视频展示修复 → 开 PR → 回复审查反馈 → 修复构建失败 → 仅在需要判断时升级给人类 → 合并。

## 关键实践

### AGENTS.md 的正确写法

```
# 错误：把所有规则塞进去（几百行）
## 正确：~100行，只放入口和指针
- 项目概述（2-3句）
- 架构地图（各领域在哪、怎么分层）
- 关键约束（必须遵守的几条）
- 文档指针（详细文档在 docs/ 的哪）
```

### 架构分层的严格执行

```
每个业务领域：
Types → Config → Repo → Service → Runtime → UI
         ↑ 只能向右依赖

跨领域：通过 Providers 接口
其他：不允许，机械执行
```

### 技术债清理循环

```
编码黄金原则 → 定期扫描 → 开重构 PR → 快速审查 → 自动合并
                                        ↑
                                    人类品味捕获一次
                                    持续执行每行代码
```

## 与其他编程范式的对比

它们不是同一个问题的不同答案，而是**同一条路上的不同阶段**。

```
手写代码 → Prompt Engineering → Vibe Coding → Harness Engineering
  我来写     我教你写            你随便写      我给你搭环境让你写对
```

从 Vibe Coding 变到 Harness Engineering，变化在哪？

Vibe Coding 说："我描述想要什么，AI 写代码，我大致看看就收了。" 问题在于：当 AI 产出量级变大，"大致看看"就不够了。Agent 每天产出几个 PR，人类审查不过来，质量开始失控。

Harness Engineering 的回应是：别审查了，让系统保证质量。不是你一个个看代码，而是你搭好护栏，让 Agent 在护栏里跑。护栏包括：架构约束（lint 强制）、上下文管理（Agent 知道在哪找信息）、可观测性（Agent 能自己验证）、自动审查（Agent 之间互相审）、技术债清理（持续自动回收）。

| 维度 | Prompt Engineering | Vibe Coding | Harness Engineering |
|------|--------------------|-------------|---------------------|
| 人类做什么 | 精心设计 prompt | 随意描述意图 | 搭环境、定约束、建反馈循环 |
| AI 做什么 | 回答问题/生成片段 | 写完整代码 | 全流程：设计到部署 |
| 质量保证 | 人工审查 | 大致看看 | 系统内建（lint/测试/Agent 审查） |
| 适用规模 | 单次交互 | 小项目 | 百万行级产品 |
| 对 AI 的信任度 | 中（人为主） | 高（信任 AI） | 高但有护栏 |

## 已知限制与开放问题

OpenAI 自己承认的不确定点：

1. **长期架构一致性**：完全 Agent 生成的代码库，几年后架构一致性怎么演变？还不知道
2. **模型能力变化**：随着模型变得更强大，这套系统怎么演进？不确定
3. **泛化性**：文章明确说"这套做法依赖特定的仓库结构和工具投入，不应假设能直接通用"
4. **人类判断力的杠杆**：人类判断在哪加杠杆最大？怎么把判断编码成能复利的形式？还在探索

### Q1 [选择题] Redis SDS 相比 C 字符串，以下哪个不是 SDS 的优势？

A. O(1) 获取字符串长度
B. 二进制安全，可以存储任意数据
C. 线程安全，多线程并发修改不需要加锁
D. 空间预分配和惰性释放减少内存分配次数

- **正确答案**：C
- **评分要点**：选择了正确选项即通过。SDS 并不是线程安全的，Redis 的单线程模型保证了线程安全，不是 SDS 本身的特性
- **知识点**：SDS 底层实现

### Q2 [选择题] 关于 Redis 持久化，以下哪个说法正确？

A. RDB 的 `save` 命令是生产环境推荐的方式
B. AOF 的 `always` 策略是最常用的生产配置
C. 混合持久化在 AOF 文件中嵌入 RDB 格式的快照数据
D. AOF 重写会阻塞主线程

- **正确答案**：C
- **评分要点**：选择了正确选项即通过。A 错（save 阻塞主线程），B 错（生产一般用 everysec），D 错（AOF 重写用子进程不阻塞）
- **知识点**：持久化策略

### Q3 [选择题] 缓存击穿的解决方案中，"逻辑过期"的含义是什么？

A. 给 key 设置物理过期时间，到期自动删除
B. 不设置 TTL，在 value 中存储过期时间，发现过期后异步更新
C. 用分布式锁控制只有一个请求去更新缓存
D. 用布隆过滤器拦截不存在的 key

- **正确答案**：B
- **评分要点**：选择了正确选项即通过。逻辑过期是不设 TTL，在 value 里存过期时间，过期后异步更新，旧数据继续用
- **知识点**：缓存击穿解决方案

### Q4 [简答题] 解释 Redis 渐进式 rehash 的过程，为什么需要渐进式？

- **参考答案**：
  1. Redis 维护两个哈希表 ht[0] 和 ht[1]
  2. 每次 CRUD 操作时，顺便将 ht[0] 中的一个桶搬迁到 ht[1]
  3. 查找时先查 ht[0]，没找到再查 ht[1]；新增只写 ht[1]
  4. 全部搬迁完成后，ht[1] 变成 ht[0]
  5. 需要渐进式的原因：一次性 rehash 大量数据会导致服务长时间阻塞，渐进式把开销分摊到多次操作中
- **评分要点**：
  1. 必须提到两个哈希表
  2. 必须提到分批搬迁（不是一次搬完）
  3. 必须说明查找/新增的规则
  4. 加分项：解释为什么不能一次性搬迁（阻塞问题）
- **知识点**：hashtable 底层实现

### Q5 [简答题] bgsave 是如何做到不阻塞主线程的？COW 机制在其中起什么作用？

- **参考答案**：
  1. bgsave 通过 fork 系统调用创建子进程
  2. fork 时，子进程复制父进程的页表，但共享同一块物理内存（不复制数据）
  3. 子进程遍历内存生成 RDB 文件，主线程继续处理请求
  4. 当主线程要修改某个数据页时，操作系统触发 COW，只复制被修改的那一页
  5. 子进程看到的是 fork 时刻的内存快照，不受主线程后续修改影响
- **评分要点**：
  1. 必须提到 fork 子进程
  2. 必须提到 COW（Copy On Write）
  3. 必须说明子进程看到的是 fork 时刻的快照
  4. 加分项：提到只有被修改的页才会被复制
- **知识点**：RDB 持久化原理

### Q6 [简答题] 描述 Redlock 算法的加锁流程，它解决的是什么问题？

- **参考答案**：
  1. Redlock 解决的是单节点 Redis 主从切换导致锁丢失的问题
  2. 部署 5 个独立 Redis 实例（无主从关系）
  3. 加锁流程：记录时间 T1 → 向 5 个实例依次发送 SET NX PX → 计算 → 判断是否 ≥ 3 个成功且耗时 < 过期时间
  4. 解锁时向所有实例发送解锁请求
  5. 多数派机制保证即使少数节点故障，锁仍然有效
- **评分要点**：
  1. 必须说明 Redlock 解决的问题（主从切换丢锁）
  2. 必须提到多数派（N/2+1）
  3. 必须提到 5 个独立实例
  4. 加分项：提到 Redlock 的争议（时钟跳变 / fencing token）
- **知识点**：分布式锁 Redlock

### Q7 [简答题] 解释布隆过滤器的原理，为什么它适合解决缓存穿透问题？它的局限性是什么？

- **参考答案**：
  1. 布隆过滤器是一个位数组 + 多个哈希函数
  2. 加入元素时：用多个哈希函数算出多个位置，全部置为 1
  3. 查询元素时：同样算出多个位置，全为 1 则"可能存在"，有 0 则"一定不存在"
  4. 适合解决穿透：拦截不存在的 key，不会误杀（说不存在一定不存在），省内存
  5. 局限性：有误判率（说存在可能不存在），不支持删除（可用 Counting Bloom Filter）
- **评分要点**：
  1. 必须提到"一定不存在"和"可能存在"两种判断
  2. 必须说明为什么适合缓存穿透场景
  3. 加分项：提到误判率和不支持删除
- **知识点**：缓存穿透解决方案

### Q8 [简答题] Redis Sentinel 的故障转移过程中，如何选择新的主节点？

- **参考答案**：
  1. 首先排除已下线或断线的从节点
  2. 按 replica-priority 最低的优先（管理员手动配置）
  3. 相同优先级选复制偏移量最大的（数据最新，与旧主节点数据差异最小）
  4. 都一样就选 run_id 最小的（运行时间最久，最稳定）
- **评分要点**：
  1. 必须提到三个选择标准的优先顺序
  2. 必须解释复制偏移量的含义（数据最新）
  3. 加分项：说明为什么 run_id 是最后的兜底条件
- **知识点**：Sentinel 故障转移

### Q9 [对比题] 对比缓存雪崩、穿透、击穿三种问题的区别（原因、现象、核心解决方案）

- **参考答案**：

  | 维度 | 雪崩 | 穿透 | 击穿 |
  |------|------|------|------|
  | 原因 | 大量 key 同时过期或 Redis 宕机 | 查询不存在的数据 | 热点 key 过期 |
  | 现象 | 大量请求打到 DB | 请求每次都穿透到 DB | 瞬时大量请求打到 DB |
  | 核心方案 | 过期时间加随机值 + 高可用 | 布隆过滤器 + 缓存空对象 | 互斥锁 + 逻辑过期 |

- **评分要点**：至少指出 3 个区别维度，能说清三种问题的不同场景
- **知识点**：缓存三大问题

### Q10 [对比题] 对比 Redis Sentinel 和 Redis Cluster 的区别，什么场景下该用哪个？

- **参考答案**：

  | 维度 | Sentinel | Cluster |
  |------|----------|---------|
  | 数据分片 | 不支持 | 支持（16384 slot） |
  | 高可用 | 支持（自动故障转移） | 支持 |
  | 最少节点 | 3 个 Sentinel + 主从 | 6 个（3 主 3 从） |
  | 适用场景 | 数据量小、只需高可用 | 数据量大、需要分片 |

- **评分要点**：至少指出 3 个区别维度，能根据场景做出正确选择
- **知识点**：高可用架构选型

### Q11 [应用题] 实际项目中，消息队列的消费端需要保证幂等性（同一条消息不重复处理）。用 Redis 设计一个幂等消费方案，要求：(1) 处理速度要快 (2) 记录至少保留 24 小时 (3) Redis 挂了怎么办

- **参考答案**：
  1. 方案：`SET consumed:{msg_id} "1" NX EX 86400`，成功则处理消息，失败则跳过
  2. 速度快：单次 Redis SET 操作，亚毫秒级
  3. 保留 24 小时：EX 86400 设置过期时间
  4. Redis 挂了：降级到数据库幂等表（INSERT ... ON DUPLICATE KEY IGNORE），或者短暂拒绝消费（宁可消息堆积也不重复处理）
- **评分要点**：
  1. 正确使用 SET NX EX
  2. 给出 Redis 不可用时的降级方案
  3. 加分项：提到消息积压与重复消费的权衡
- **知识点**：分布式锁应用 / 幂等消费

### Q12 [应用题] 你的系统有一个热点商品详情接口，QPS 10 万。缓存 key 是 `product:{id}`，过期时间 1 小时。在某个时刻，大量请求突然打到数据库。分析可能的原因并给出完整解决方案。

- **参考答案**：
  1. 可能原因一：缓存击穿（热点 key 过期）。方案：互斥锁或逻辑过期
  2. 可能原因二：缓存雪崩（大量 key 同时过期）。方案：过期时间加随机值
  3. 可能原因三：Redis 宕机。方案：Sentinel 高可用 + 本地缓存兜底
  4. 完整方案：逻辑过期（热点数据不设 TTL）+ 布隆过滤器 + Sentinel + 本地缓存（Caffeine 二级缓存）+ 限流降级
- **评分要点**：
  1. 能区分击穿和雪崩
  2. 给出至少 2 种解决方案
  3. 加分项：提到多级缓存和限流
- **知识点**：缓存三大问题综合应用

### Q13 [简答题] Redis 跳表（skiplist）的查找时间复杂度是多少？为什么 Redis 选跳表而不是红黑树来实现 ZSet？

- **参考答案**：
  1. 查找时间复杂度：O(logN)，和平衡树相当
  2. 选跳表的原因：
     - 实现简单，代码量小，容易理解和维护
     - 范围查询方便：找到起点后沿 Level 0 链表往后遍历即可，红黑树需要中序遍历
     - 内存占用可调：通过调整概率参数控制层数
     - 并发友好：锁粒度可以做到更细
  3. Redis 作者 Antirez 的原话大意：跳表和平衡树性能相当，但跳表更简单，范围操作更自然
- **评分要点**：
  1. 必须说对时间复杂度 O(logN)
  2. 至少给出 2 个选跳表的理由
  3. 加分项：提到范围查询的优势
- **知识点**：skiplist 底层实现

# 微服务架构 — AI 笔记

> 来源：Martin Fowler《Microservices》（2014）、Chris Richardson microservices.io 模式目录、Microsoft Azure 微服务架构指南
> 生成时间：2026-04-26
> 学习目标：面试准备，理解核心概念和设计决策

---

## 锚点

一个电商系统，用户点"下单"，背后要干三件事：扣库存、扣款、发物流通知。在单体应用里，这三件事在同一个进程里、同一个数据库事务里，要么全成功要么全回滚。

现在把它们拆成三个独立服务，跑在三个进程里。库存服务挂了，钱已经扣了怎么办？物流服务慢了，用户等多久？三个服务各用各的数据库，数据不一致了怎么发现？

**微服务所有的问题，都从这个拆分开始。** 拆了之后怎么通信、怎么保证不出错、出了错怎么定位、怎么部署——每个问题催生了一整套解决方案。

## 核心问题

为什么要把好好的单体应用拆成几十个服务？因为单体应用长大了会出问题：改一行代码要重新部署整个系统，扩展要扩展整个应用而不是瓶颈部分，团队大了改同一份代码互相踩脚。但拆了之后，通信、一致性、可靠性全变难了。**微服务不是免费的午餐，是用运维复杂度换开发灵活度。**

## 核心概念

### 单体 vs 微服务

单体应用：所有代码打包成一个可执行文件，跑在一个进程里。简单、好调试、好部署。但代码量一大，编译变慢、部署变慢、改一个小功能要把整个应用重启一遍。团队协作也难，十个人改同一个代码仓库，互相等。

微服务：把应用拆成一组小服务，每个服务跑在自己进程里，通过 HTTP 或消息队列通信。每个服务可以独立部署、独立扩展、用不同的技术栈。代价是运维复杂度飙升——你不再是管一个应用，而是管十几个。

Martin Fowler 本人的建议：**先从单体开始，保持模块化，等单体成为问题再拆。** 一开始就上微服务，大概率是过度设计。

### 服务拆分

拆分的依据是什么？两种主流思路：

1. **按业务能力拆**（Martin Fowler 推荐）：围绕业务领域组织服务。比如电商系统拆成订单服务、库存服务、用户服务、支付服务。每个服务覆盖一个完整的业务功能，从前端到数据库。"一个团队负责一个业务"。

2. **按子域拆**（DDD 的 Bounded Context）：用领域驱动设计的方法，先画出业务领域的边界，每个边界对应一个服务。和"按业务能力"很像，但更强调领域建模，适合业务复杂的系统。

拆分的一个经验法则：**经常一起变的东西放一起，不一起变的拆开。** 如果你发现自己每次改代码都要同时改两个服务，说明它们不该拆开。如果两个服务一个月都没互相影响，说明拆对了。

拆太细的代价：服务之间调用链变长、调试变难、部署变复杂。一个极端的例子：把"用户注册"拆成"验证邮箱"、"创建账号"、"发欢迎邮件"三个服务——这拆过头了，它们本来就是一个业务流程。

### 服务间通信

服务拆开了，它们之间怎么说话？两条路：同步调用和异步消息。

**同步调用**（HTTP REST / gRPC）：服务 A 直接调用服务 B 的接口，等 B 返回结果再继续。简单直接，像打电话——对方不接你就卡住了。问题是如果 B 慢了或者挂了，A 也被拖慢甚至拖死。调用链长了（A→B→C→D），任何一环出问题，整条链都受影响。

**异步消息**（Kafka / RabbitMQ / RocketMQ）：服务 A 把消息丢到队列里就走了，服务 B 从队列里取消息处理。像发邮件——发完就干别的事去了。好处是解耦，A 不关心 B 是不是在线、处理快不快。坏处是调试难，消息丢了、重复了、顺序错了都不好排查。

怎么选？**需要立即拿到结果的用同步，不需要的用异步。** 比如"查询用户信息"必须同步返回，"发送订单确认邮件"可以异步。实际系统中两种方式混用，用消息队列做服务之间的解耦，用同步调用做需要即时响应的查询。

### API 网关

客户端（手机 App / 网页）要调多个服务，总不能让客户端知道每个服务的地址吧？API 网关就是所有请求的统一入口。

它做的事：路由（把请求转发到对应的服务）、认证（统一验证 token）、限流（防止某个客户端打爆服务）、聚合（一个请求需要多个服务的数据，网关负责组合）。

一个变种是 **BFF（Backend for Frontend）**：给每种客户端一个专属的网关。手机 App 一个网关、网页一个网关，因为它们需要的数据格式、字段可能不一样。避免一个网关承担所有客户端的逻辑变得臃肿。

### 数据库去共享

微服务的一个核心原则：**每个服务有自己的数据库，不共享。** 为什么？共享数据库 = 共享内存 = 耦合。两个服务写同一张表，改表结构要协调两个团队，失去了独立部署的意义。

但这带来一个难题：跨服务的数据一致性。单体应用里一个事务就能搞定的事，现在要跨多个数据库。解决方案是 **Saga 模式**：把一个大事务拆成一系列小事务，每个服务做完自己的事后发一个事件，下一个服务收到事件继续做。如果中间某一步失败了，发一个补偿事件，把之前做的事撤回。

比如下单流程：订单服务创建订单 → 发事件 → 库存服务扣库存 → 发事件 → 支付服务扣款。如果扣款失败，支付服务发补偿事件 → 库存服务加回库存 → 订单服务取消订单。这比数据库事务复杂得多，但是微服务下的标准做法。

这种一致性叫 **最终一致性**（Eventual Consistency）：不是每时每刻都一致，但过一会儿就一致了。大部分业务场景能接受。

### 熔断器（Circuit Breaker）

服务 A 调服务 B，B 挂了，A 不应该一直傻等。熔断器的工作方式跟电路保险丝一样：正常时请求正常通过（关闭状态）；如果连续失败超过阈值，熔断器"跳闸"（打开状态），后续请求直接返回错误，不再调用 B；过一段时间后放一个请求试试（半开状态），如果成功就恢复（关闭），失败继续断开。

为什么重要？**防止故障级联。** B 挂了，A 的线程全卡在等 B，A 也挂了，调用 A 的 C 也挂了——雪崩效应。熔断器让 A 在 B 出问题时快速失败，保护自己不被拖死。

还有一个相关概念叫 **舱壁隔离**（Bulkhead）：像船的防水舱一样，给不同的服务调用分配独立的线程池或连接池。一个服务出问题只影响它自己的池子，不会把所有资源耗尽。

### 幂等性

幂等 = 同一个操作执行一次和执行多次，效果一样。为什么在微服务中特别重要？

因为网络不可靠。服务 A 调服务 B，请求发出去了，B 处理完了但响应在网络中丢了。A 没收到响应，以为失败了，重试。B 又处理了一遍。如果 B 是"扣款 100 元"，扣了两次就出事了。如果 B 是幂等的（"设置余额为 X"或者"用唯一 ID 去重"），重多少次都一样。

实现幂等的常见方式：
- **唯一请求 ID**：每个操作带一个全局唯一 ID，服务端记住处理过的 ID，重复的直接返回之前的结果
- **乐观锁**：带版本号更新，版本号不对就拒绝
- **数据库唯一约束**：利用唯一索引防止重复插入

### 分布式链路追踪

一个请求从网关进来，调了服务 A，A 调了 B，B 调了 C，C 查了数据库。请求慢了，到底是哪一步慢？单体应用里看日志就行，微服务里日志散落在不同机器上。

链路追踪的做法：给每个请求分配一个 **Trace ID**，请求每经过一个服务，就带着这个 ID。每个服务记录自己这段处理的耗时和状态（叫一个 Span）。最后把所有 Span 按 Trace ID 串起来，就能看到完整的调用链和每一步的耗时。

像快递追踪：一个包裹从仓库 → 分拣中心 → 运输车 → 派送站 → 你手上，每一步都扫条码记录时间。Trace ID 就是那个条码。常用工具：Jaeger、Zipkin、SkyWalking。

### 配置管理

每个服务都有自己的配置：数据库地址、第三方 API 密钥、超时时间。几十个服务，配置文件散落各处，改一个数据库密码要改几十个地方。

两种方式：

1. **环境变量**：简单，服务启动时从环境变量读配置。适合少量配置，改了要重启服务。容器化环境（Docker/K8s）天然支持。

2. **配置中心**（Nacos / Apollo / Spring Cloud Config）：所有配置集中存放在一个地方，服务启动时去配置中心拉取。好处是改配置不用重启（热更新），还能做版本管理和灰度发布。适合配置多、需要动态调整的场景。

怎么选？配置少、不常变用环境变量。配置多、需要热更新用配置中心。实际中经常混用：基础配置（端口、日志级别）用环境变量，业务配置（功能开关、阈值）用配置中心。

## 与其他架构的对比

微服务、单体、SOA 三者不是"哪个更先进"的关系，是适用场景不同。

**单体**：简单，开发快，调试方便。适合小团队、早期产品、业务逻辑不复杂的系统。问题出在"大"了之后。

**SOA**（面向服务架构）：微服务的前身，也把系统拆成服务，但倾向于用重量级的 ESB（企业服务总线）做服务间通信和编排。ESB 本身成了瓶颈和单点。微服务可以看作 SOA 的"轻量级版本"——去掉了 ESB，用更简单的通信方式。

**微服务**：灵活，可独立部署和扩展，适合大团队、高并发、业务复杂的系统。但运维成本高，需要成熟的 DevOps 和监控体系。

| 维度 | 单体 | SOA | 微服务 |
|------|------|-----|--------|
| 拆分粒度 | 不拆 | 粗粒度服务 | 细粒度服务 |
| 通信 | 进程内调用 | ESB（重量级） | HTTP/MQ（轻量级） |
| 数据库 | 共享 | 倾向共享 | 每个服务独立 |
| 部署 | 整体部署 | 较独立 | 独立 |
| 运维复杂度 | 低 | 中 | 高 |
| 适用场景 | 小团队/早期 | 企业集成 | 大规模系统 |

## 面试高频考点

| 考点 | 他真正想知道的 | 回答要点 | 常见错误 |
|------|---------------|----------|----------|
| 微服务和单体优劣 | 你能不能根据场景做选择 | 单体简单好上手但扩展受限，微服务灵活但运维复杂。关键是"什么时候该拆"而不是"哪个更好" | 一上来就说微服务更好 |
| 什么时候不该用微服务 | 你有没有工程判断力 | 团队小（<10人）、业务不复杂、没有成熟的 DevOps 体系、不需要独立扩展 | 为了"技术先进"而用微服务 |
| 服务怎么拆 | 你理解拆分的原理 | 按业务能力或 DDD 子域拆。一起变的一起放，不一起变的拆开。拆太细和拆太粗都有问题 | 按技术层拆（UI 团队、数据库团队） |
| 同步 vs 异步通信 | 你理解两种通信的取舍 | 需要即时结果用同步，解耦和削峰用异步。实际混用。同步的问题：故障级联；异步的问题：调试难 | 只用一种方式 |
| API 网关 | 你知道微服务的入口设计 | 统一入口：路由、认证、限流、聚合。BFF 是变种，给不同客户端专属网关 | 以为网关就是反向代理 |
| 熔断器 | 你理解分布式系统的容错 | 像保险丝：正常→连续失败→跳闸→过会试一下→恢复。防止故障级联 | 不知道为什么需要熔断 |
| 幂等性 | 你理解分布式环境下的重试问题 | 网络不可靠导致重试，非幂等操作会重复执行。用唯一 ID / 乐观锁 / 唯一约束保证幂等 | 以为数据库事务能解决一切 |
| 分布式链路追踪 | 你排查过微服务的问题 | Trace ID 贯穿调用链，Span 记录每段耗时。把散落各服务的日志串起来 | 不知道 Trace ID 怎么传递 |
| 配置管理 | 你理解微服务的运维实践 | 少量配置用环境变量，大量配置和热更新用配置中心。实际混用 | 只知道一种方式 |
| 数据一致性 | 你理解分布式事务的难题 | 跨服务不能用数据库事务。用 Saga 模式：拆成小事务 + 补偿操作。最终一致性 | 还在用分布式事务（2PC） |

## Martin Fowler 的九大特征

Martin Fowler 在 2014 年的文章中总结了微服务架构的九个常见特征。这不是一个清单必须全部满足，而是"大部分微服务架构会体现大部分特征"：

1. **通过服务组件化** — 服务是独立替换、独立升级的组件，不是库
2. **按业务能力组织** — 按业务领域而不是技术层来拆分团队和服务
3. **做产品不做项目** — 团队对服务全生命周期负责（"you build, you run it"）
4. **智能端点、傻瓜管道** — 业务逻辑在服务里，通信管道只管传输不管逻辑。反 ESB
5. **去中心化治理** — 不强制统一技术栈，每个服务选最合适的工具
6. **去中心化数据管理** — 每个服务管理自己的数据库，接受最终一致性
7. **基础设施自动化** — CI/CD 是基本要求，部署必须自动化
8. **容错设计** — 服务随时可能挂，系统要能容忍失败。监控和快速恢复是必需品
9. **演进式设计** — 服务边界可以随业务演进而调整，甚至整个服务可以被废弃重写

一个关键的观察：Fowler 最后的结论是谨慎乐观的。他说微服务还太新，没有足够的长期数据证明它比单体更好。而且他建议**不要一开始就用微服务，先从模块化的单体开始，等有问题再拆**。

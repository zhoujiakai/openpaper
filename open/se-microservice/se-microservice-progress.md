## 已掌握的关键知识点

### Section 1: 单体 vs 微服务
- 单体应用长大了会出问题：部署耦合、扩展浪费、团队冲突
- 微服务用运维复杂度换开发灵活度
- Fowler 建议：先从模块化单体开始，有问题再拆

### Section 2: 服务拆分
- 按业务能力拆，不按技术层拆
- 围绕核心对象分组，但还要看变动频率和技术依赖是否一致
- 用户能正确拆分外卖系统（5 个服务），在"订单+支付是否合并"上有讨论空间

### Section 3: 服务间通信
- 同步：简单但有调用链风险和雪崩效应
- 异步：解耦但调试难
- 判断标准：用户在等就同步，不需要马上知道就异步
- 用户能正确区分外卖流程中的同步/异步步骤

### Section 4: API 网关
- 统一入口：路由、认证、限流、聚合
- 没有网关的问题：客户端记不住地址、无法动态扩展、每个服务重复实现认证、业务逻辑散落前后端
- BFF 是变种：给不同客户端专属网关
- 用户能正确分析无网关场景下的问题（扩展性、认证冗余、请求聚合复杂度）

### Section 5: 数据库去共享
- 每个服务有自己的数据库，不共享——共享就是耦合
- 跨服务数据一致性用 Saga 模式（补偿事件，不是回滚）
- 最终一致性：不是每时每刻一致，但过一会儿就一致了
- 不能接受最终一致性的场景：金融余额等对数据准确性有实时要求的场景
- 用户纠正：理解了"预留机制"和"Saga"是两个不同的概念

### Section 6: 熔断器
- 三种状态：关闭（正常）→ 打开（直接失败）→ 半开（试探恢复）
- 核心目的：牺牲少数请求体验，保护整个系统不雪崩
- 舱壁隔离：给不同服务调用分配独立线程池
- 用户理解到位：网络波动本身也是需要预防的异常

### Section 7: 幂等性
- 同一个操作做一次和做多次效果一样
- 网络不可靠导致重试，非幂等操作会重复执行
- 实现方式：唯一请求 ID、乐观锁、数据库唯一约束
- 用户用"健忘症患者吃药打卡"类比，准确对应唯一请求 ID 的思路

### Section 8: 分布式链路追踪
- Trace ID 贯穿调用链，Span 记录每段耗时
- 通过 HTTP 请求头透传（如 X-Trace-Id）
- 框架层自动处理，业务代码基本不用管
- 常用工具：Jaeger、Zipkin、SkyWalking

### Section 9: 配置管理
- 环境变量：简单，改了要重启
- 配置中心：集中管理，支持热更新和版本管理
- 混用：基础配置用环境变量，业务配置用配置中心
- 配置中心挂了：各服务用本地缓存继续跑，不影响运行

## 被纠正过的误区

- 用户最初认为"支付"可以异步处理。实际上支付通常是同步的——用户等着看结果，必须立刻返回成功或失败。区分"用户在等"和"系统在等"是关键
- 银行扣款的并发问题是单服务内部问题（用数据库锁解决），不是跨服务问题。跨服务问题应该是"支付服务扣款成功但优惠券服务标记失败"
- 用户一度混淆"链路追踪"和"Saga"——链路追踪是监控工具，Saga 是事务协调模式
- 用户最初将"预留机制"和"Saga"混为一谈——预留解决资源唯一性，Saga 解决跨服务协调

> - 说明：写给有 Python 基础、想系统掌握 C 语言的人。默认你已经知道什么是变量、循环、函数——这里只讲 C 不一样的地方。
>

## 锚点：从租房到买房

Python 像一个全包公寓。水电煤气有人管，垃圾有人收，你只管住。东西找不到了？物业有监控（GC 自动回收）。想砸墙改造？不行，承重墙不能动。

C 像买了一套毛坯房。水管自己接，电路自己布，垃圾自己倒（手动 free）。但你可以砸任何一面墙，改任何一条线，甚至在客厅装个车床——只要你付得起代价。

这个比喻会贯穿全文。

## 1. 你的代码怎么变成可执行文件的

### Python 怎么跑的

```
你写 .py 文件 → Python 解释器逐行读 → 翻译成字节码 → 虚拟机执行
```

整个过程是**运行时**完成的。语法错误？跑到那一行才报。

### C 怎么跑的

```
你写 .c 文件 → 预处理 → 编译 → 汇编 → 链接 → 得到可执行文件
```

四个阶段，一步不能少：

| 阶段 | 干什么 | 产物 |
|------|--------|------|
| **预处理** | `#include` 展开头文件，`#define` 替换宏 | `.i` 文件（纯 C 代码） |
| **编译** | C 代码翻译成汇编 | `.s` 文件 |
| **汇编** | 汇编翻译成机器码 | `.o` 文件（二进制） |
| **链接** | 把多个 `.o` 和库拼成一个可执行文件 | `a.out` / `.exe` |

关键区别：**C 的错误在编译阶段就报**，不是运行时。类型写错、少分号——你都跑不起来。

另外，C 编译器不会帮你处理依赖。你得手动告诉它去哪找头文件（`-I`）、链接哪个库（`-l`）。

### 最小的 C 程序

```c
#include <stdio.h>   // 预处理：把 stdio.h 的内容复制过来

int main() {
    printf("hello\n");
    return 0;
}
```

每一行都有意义：
- `#include <stdio.h>` — 告诉编译器 "printf 函数长什么样"，但不包含函数的实现代码。实现在链接阶段才合进来。
- `int main()` — C 程序的入口，`int` 是返回类型。返回 0 表示正常，非 0 表示异常。
- `return 0;` — 分号不是装饰，是语句结束标志。忘了写编译器报错。

编译运行：
```bash
gcc hello.c -o hello
./hello
```

## 2. 类型系统：C 要你提前画好图纸

### Python 的做法

```python
x = 42       # Python 自动推断：这是个 int，占 28 字节
x = "hello"  # 现在 x 变成 str 了，没问题
```

Python 的变量是**标签**，贴在对象上。对象自带类型信息。运行时随时换。

### C 的做法

```c
int x = 42;      // 明确声明：x 是 int，占 4 字节，存整数
x = "hello";     // 编译报错！类型不匹配
```

C 的变量是**内存盒子**。盒子的尺寸在编译时就定死了，之后不能变（能变的是盒子里的值）。

### C 的基本类型及尺寸

| 类型 | 典型大小（64位） | 含义 |
|------|-----------------|------|
| `char` | 1 字节 | 字符或小整数 |
| `short` | 2 字节 | 短整数 |
| `int` | 4 字节 | 标准整数 |
| `long` | 8 字节 | 长整数 |
| `long long` | 8 字节 | 很长整数 |
| `float` | 4 字节 | 单精度浮点 |
| `double` | 8 字节 | 双精度浮点 |

**注意**：上面写的是"典型大小"。C 标准只规定了最小范围，不规定具体字节数。`int` 可以是 2 字节也可以是 4 字节。`sizeof()` 运算符能查实际大小。

### signed vs unsigned

Python 的 `int` 可以无限大（自动扩）。C 的 `int` 是固定位数的 bit 序列。

最高位被用来表示正负号——`signed`（默认）的时候。最高位为 1 的数是负数（补码表示）。

```c
unsigned int a = 4294967295;  // 32 位全 1，无符号：最大正数
int b = 2147483647;           // 32 位，最高位 0，有符号最大正数
```

> 补码不是必须现在记住的东西。先知道：有符号数的范围是 `[-2^(n-1), 2^(n-1)-1]`，无符号是 `[0, 2^n-1]`。

### const 和 volatile

两个让编译器听话的关键字：

```c
const int x = 10;    // x 的值不能改（编译期检查）
x = 20;              // 编译报错
```

```c
volatile int flag;   // 告诉编译器：别优化它，每次用都从内存重新读
// 用在：硬件寄存器、中断里修改的变量、多线程共享变量
```

`volatile` 在用不到的场景里先忽略，但心里留个印象——嵌入式里很常见。

## 3. 指针：C 语言的分水岭

这是 C 和 Python 差距最大的地方，也是你这次学习的核心。

### 说人话：指针是什么

**指针就是一个存着"门牌号"的变量。**

Python 里你不需要知道变量在内存里的地址——那是物业的事。C 里你可以直接拿到地址、存起来、传给函数、甚至自己修改地址。

打个比方：变量是房子，值就是住在房子里的人。指针不是房子，是一张写着门牌号的纸条。拿着这张纸条，你能找到那栋房子（解引用），也能把纸条交给别人（指针赋值）。

### 两个核心运算符

| 运算符 | 作用 | 类比 |
|--------|------|------|
| `&` 取地址 | 得到变量的内存地址 | "告诉我你家的门牌号" |
| `*` 解引用 | 拿到地址对应的值 | "按门牌号去拜访" |

```c
int x = 42;
int *p = &x;   // p 是一张纸条，上面写着 x 的地址

printf("%p\n", p);   // 打印地址（一串十六进制，比如 0x7ffd...）
printf("%d\n", *p);  // 解引用，打印 42

*p = 100;            // 通过纸条修改房子里的内容
printf("%d\n", x);   // 输出 100
```

### 为什么需要指针？

**原因一：函数需要修改外部变量**

Python 里，函数内 `x = x + 1` 可能修改外部变量（如果 x 是可变对象），也可能不修改（如果 x 是不可变对象）。规则取决于对象类型。

C 没有这种暧昧。函数参数永远是**值传递**——函数拿到的是拷贝：

```c
void add_one(int n) {
    n = n + 1;      // 只改了拷贝，外面的原值不变
}
```

想真正修改函数外变量的值？传地址：

```c
void add_one(int *p) {
    *p = *p + 1;     // 通过门牌号找到原房子，直接改
}

int x = 5;
add_one(&x);         // 传门牌号过去
// x 现在是 6
```

**原因二：避免大量数据拷贝**

一个 10000 个元素的数组，传值的话要复制全部数据。传指针只需复制一个地址（8 字节）。

### 指针声明怎么读

```c
int *p;       // p 是指向 int 的指针
int **pp;     // pp 是指向 (指向 int 的指针) 的指针（二级指针）
int *arr[10]; // arr 是含 10 个元素的数组，每个元素是指向 int 的指针
int (*pa)[10];// pa 指向含 10 个 int 的数组
```

读声明有个口诀：**从变量名出发，从内向外，先右后左。**

- `int *p`：p（从内）→ 右边没了 → 看左边是 `*` 指针 → 指向 `int`
- `int *arr[10]`：arr → 右边有 `[10]`，是数组 → 数组里是 `*` 指针 → 指向 `int`
- `int (*pa)[10]`：\*pa（pa 是指针）→ 指向 `int[10]`（含 10 个 int 的数组）

### NULL 指针

```c
int *p = NULL;  // 空指针，不指向任何有效地址
```

`NULL` 是一个特殊的地址值（通常是 `(void*)0`）。解引用 NULL 会导致**段错误 (Segmentation Fault)**——程序直接崩溃。这算是一种保护机制：宁可崩溃让你知道，也不让你读写不该读写的地方。

每次 `malloc` 分配内存后、解引用指针前，都要检查是否为 NULL。

### 常见错误

1. **野指针**：声明了指针但没初始化，里面是垃圾值。解引用行为未定义。
2. **悬垂指针**：`free` 之后还拿着原来的地址。那个地址可能已经被分配给别的变量了。
3. **类型不匹配**：把 `int*` 的值赋给 `char*`，编译器会警告。

## 4. 数组：一扇永远开着的后门

### 数组的本质

C 的数组就是**连续排列的同一类型元素**。没有长度信息，没有边界检查。你给个数组，C 只记住起始地址。

```c
int arr[5] = {10, 20, 30, 40, 50};
```

在内存里是这样的（假设地址是 0x1000）：

```
地址:  0x1000  0x1004  0x1008  0x100C  0x1010
值:      10      20      30      40      50
       arr[0]  arr[1]  arr[2]  arr[3]  arr[4]
```

每个 int 占 4 字节，所以相邻元素地址差 4。

### 数组名就是指针（嗯，差不多）

```c
int arr[5] = {10, 20, 30, 40, 50};
int *p = arr;     // 等价于 int *p = &arr[0];
```

数组名 `arr` 在大多数场景下会**退化成**指向首元素的指针。所以：

```c
arr[2]  等价于  *(arr + 2)    // 从首地址往后移 2 个元素
p[2]    等价于  *(p + 2)
```

这就是为什么 `arr[2]` 和 `2[arr]` 都合法——有点邪门，别用。

### 数组名和指针的区别

不同：

```c
int arr[5];
int *p = arr;

sizeof(arr);   // 20（整个数组的大小）
sizeof(p);     // 8（指针本身的大小，64位系统）

arr = p;       // 编译错误！数组名不是变量，不能赋值
p = arr;       // 正常，p 可以重指向
```

`arr` 知道整个数组有多大，`p` 只知道地址。一旦数组传给函数，退化立刻发生：

```c
void func(int arr[]) {
    // 这里 arr 已经是指针了
    sizeof(arr);  // 总是 8（指针大小），不是数组大小！
}

// 三种写法完全等价
void func(int arr[]);
void func(int arr[10]);  // 这里的 10 编译器会忽略，纯属文档作用
void func(int *arr);
```

所以传数组给函数时，**必须同时传长度**：

```c
void print_array(int *arr, size_t len) {
    for (size_t i = 0; i < len; i++) {
        printf("%d ", arr[i]);
    }
}
```

### 字符串：末尾有特殊标记的字符数组

C 没有字符串类型。字符串就是一个 `char` 数组，以 `\0`（ASCII 值为 0 的字符）结尾。

```c
char s[] = "hello";  // 自动分配 6 个字节：h e l l o \0
char s[6] = {'h', 'e', 'l', 'l', 'o', '\0'};  // 等价
```

`<string.h>` 里的函数（`strlen`、`strcpy`、`strcmp`）全靠 `\0` 找结尾。忘了放 `\0` 会导致这些函数一直读下去，直到碰到内存里某个随机位置的 0——缓冲区溢出的典型来源。

### Python vs C：再也不想念切片语法

```python
# Python
arr = [1, 2, 3, 4, 5]
sub = arr[1:3]   # [2, 3]，全新列表，安全又方便
```

C 里没有切片。数组越界**不会立即报错**——它可能读到垃圾值，可能覆盖其他变量的数据，也可能刚好正常工作然后某天突然崩。

```c
int arr[5] = {1, 2, 3, 4, 5};
int x = arr[10];  // 编译器不拦你，结果是什么？不知道。
```

这个东西叫做**未定义行为 (Undefined Behavior)**，简称 UB。看到 UB 要警觉——它意味着程序的任何行为都是合理的结果，包括看起来运行正常。

## 5. 动态内存：你自己当垃圾工

Python 里 `list.append()` 不担心空间不够，内存自动扩。`del` 或变量离开作用域，GC 会回收。

C 里没有 GC。堆上的内存你申请了你负责还。

### malloc、calloc、realloc、free

```c
#include <stdlib.h>

// 申请 100 个 int 的空间
int *arr = (int *)malloc(100 * sizeof(int));
if (arr == NULL) {
    // 必须检查！系统可能没内存了
    fprintf(stderr, "内存分配失败\n");
    exit(1);
}

// 申请并清零（分配 5 个 int，初始化为 0）
int *arr2 = (int *)calloc(5, sizeof(int));

// 扩容（把 arr 扩到 200 个 int）
int *tmp = (int *)realloc(arr, 200 * sizeof(int));
if (tmp == NULL) {
    // realloc 失败时原内存块还在，别直接 arr = realloc(...)
    // 要先检查
    free(arr);
    exit(1);
}
arr = tmp;

// 释放
free(arr);
arr = NULL;  // 防止悬垂指针
```

### 为什么 `sizeof(int)` 而不直接写 4？

因为不同平台上 `int` 可能是 2 字节或 4 字节。用 `sizeof` 让你的代码可移植。

### 内存泄漏：C 界的慢性病

```c
void leaky() {
    int *p = malloc(100 * sizeof(int));
    // 没有 free(p)！
    // 函数返回后 p 消失，但那 400 字节的堆内存还在——没人知道在哪了
}
```

长期运行的程序（服务器、嵌入式设备）里泄漏会慢慢吃光内存。

**铁律**：每个 `malloc` / `calloc` 必须有且只有一个对应的 `free`。

### 堆 vs 栈：别再搞混

| | 栈 (Stack) | 堆 (Heap) |
|---|-----------|-----------|
| **谁管** | 编译器自动 | 你手动 |
| **生命周期** | 函数返回自动回收 | 直到你 free |
| **大小** | 有限（通常几 MB） | 取决于系统 |
| **速度** | 快 | 慢 |
| **分配方式** | 移动栈指针 | 查找空闲块 |
| **典型用途** | 局部变量、小数组 | 动态数据结构、大数组、生命周期不确定的数据 |

**栈的黄金规则**：永远不要返回指向栈上变量的指针。

```c
int *bad() {
    int x = 42;
    return &x;  // 不行！x 在函数返回后会被回收
}

int *good() {
    int *p = malloc(sizeof(int));
    *p = 42;
    return p;   // 堆上分配的，函数返回后还在
}
```

### C 程序的内存全景

从高地址到低地址：

```
┌─────────────┐ 高地址
│    栈       │ ← 局部变量，向下增长
│    ↓       │
│            │
│    ↑       │
│    堆       │ ← malloc 来的，向上增长
├─────────────┤
│   BSS 段    │ ← 未初始化的全局/静态变量
├─────────────┤
│   数据段    │ ← 已初始化的全局/静态变量
├─────────────┤
│   代码段    │ ← 你的程序指令（只读）
└─────────────┘ 低地址
```

栈和堆对向增长，碰到一起就是栈溢出。

## 6. 结构体：C 版的 class（骨头版）

Python 的 class 可以放数据和方法，可以继承，有魔术方法。C 的 struct 只能放数据——说它是骨头，是因为只有骨架没有肉。

```c
struct Point {
    int x;
    int y;
};

struct Point p1;           // 声明变量
p1.x = 10;
p1.y = 20;

struct Point *pp = &p1;    // 指针访问用 ->
pp->x = 30;               // 等价于 (*pp).x = 30
```

### typedef 简化写法

每次写 `struct Point` 比较繁琐，用 `typedef` 起别名：

```c
typedef struct {
    int x;
    int y;
} Point;

Point p1;         // 不再需要写 struct
```

### 内存对齐：隐形的空间浪费

```c
typedef struct {
    char  a;   // 1 字节
    int   b;   // 4 字节
    char  c;   // 1 字节
} BadLayout;

printf("%zu\n", sizeof(BadLayout));  // 你可能以为是 6，实际是 12
```

对齐规则简版：每个成员的起始地址必须是其自身大小的整数倍。结构体总大小是最大成员大小的整数倍。

所以上面那个结构实际布局是：

```
a: 1字节 | padding: 3字节 | b: 4字节 | c: 1字节 | padding: 3字节  = 12字节
```

优化方法：成员按大小降序排列。

```c
typedef struct {
    int   b;   // 4 字节
    char  a;   // 1 字节
    char  c;   // 1 字节
    // 自动对齐：2 字节 padding
} GoodLayout;  // sizeof = 8
```

## 7. 函数指针：一等公民的边缘

Python 里函数是一等公民，传来传去很自然：

```python
def add(x, y): return x + y
def apply(func, a, b): return func(a, b)
apply(add, 3, 4)  # 7
```

C 也能做到，但写法有点绕：

```c
int add(int a, int b) {
    return a + b;
}

int (*func_ptr)(int, int) = &add;  // 声明一个函数指针
int result = func_ptr(3, 4);       // 通过指针调用
```

声明怎么读：`func_ptr` → 左边有 `*` 是指针 → 右边有 `()` 是函数 → 参数是两个 `int` → 返回 `int`。所以是"指向函数的指针，这个函数接收两个 int 返回一个 int"。

实际用途：
- 回调函数（比如 `qsort` 的比较函数）
- 状态机里的处理函数表
- 嵌入式中断向量表（硬件中断到来时跳转到对应的处理函数）

### qsort 示例

```c
#include <stdlib.h>

int compare(const void *a, const void *b) {
    return (*(int *)a - *(int *)b);  // 从小到大
}

int arr[] = {5, 2, 8, 1, 3};
qsort(arr, 5, sizeof(int), compare);
```

## 8. 位运算：为嵌入式铺路

Python 有位运算，但很少有人在学校里被要求写寄存器配置。C 的位运算和嵌入式紧密绑定——GPIO 的每个引脚、定时器的每个模式选择，通常都是通过位操作来配置的。

### 六个运算符

| 运算符 | 含义 | 例子 |
|--------|------|------|
| `&` | 按位与 | `0b1100 & 0b1010 → 0b1000` |
| `\|` | 按位或 | `0b1100 \| 0b1010 → 0b1110` |
| `^` | 按位异或 | `0b1100 ^ 0b1010 → 0b0110` |
| `~` | 按位取反 | `~0b0000 → 0b1111` (8位下) |
| `<<` | 左移 | `0b0001 << 2 → 0b0100` |
| `>>` | 右移 | `0b1000 >> 2 → 0b0010` |

右移分两种：
- **逻辑右移**：左边补 0（`unsigned` 专属）
- **算术右移**：左边补符号位（`signed` 可能有）

### 嵌入式常用操作模式

```c
#define BIT(n) (1 << (n))

uint32_t reg = 0;

// 设置第 3 位（从 0 开始数）
reg |= BIT(3);          // reg = reg | 0b00001000

// 清除第 3 位
reg &= ~BIT(3);         // reg = reg & 0b11110111

// 翻转第 3 位
reg ^= BIT(3);          // reg = reg ^ 0b00001000

// 读第 3 位
int bit3 = (reg >> 3) & 1;

// 设置一个多位域（比如把 bits 4-6 设为 101）
reg &= ~(0b111 << 4);   // 先清除目标位的旧值
reg |= (0b101 << 4);    // 再写入新值
```

这些都是嵌入式开发里写寄存器配置的基本套路。后面学到 STM32 的 GPIO 配置时会反复用到。

## 9. 宏与预处理器：一把双刃剑

Python 没有预处理器。C 的 `#define` 是在编译前做的**纯文本替换**，编译器根本看不到宏名。

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

每一层括号都有用。没有括号的话：

```c
#define BAD_MAX(a, b) a > b ? a : b

int x = BAD_MAX(1, 2) + 3;  // 展开后：1 > 2 ? 1 : 2 + 3
                             // 运算符优先级导致结果跟你想要的完全不同
```

**宏铁律**：宏定义里所有参数和整个表达式都加括号。或者——用 `inline` 函数代替。

```c
static inline int max(int a, int b) {
    return a > b ? a : b;
}
```

`inline` 函数有类型检查，不会出文本替换的幺蛾子。

### 条件编译

```c
#ifdef DEBUG
    printf("调试信息：x = %d\n", x);
#endif

#if defined(__linux__)
    // Linux 特定代码
#elif defined(_WIN32)
    // Windows 特定代码
#endif
```

`#ifdef` 和 `#if` 的区别：前者只检查宏是否定义（不管值），后者检查表达式的值。

### #include 的本质

`#include "myheader.h"` 就是直接把那个文件的内容**全文复制**到当前位置。没有命名空间，没有模块系统——两个头文件里定义同名的东西就会冲突。

`#include` 防重复的标准写法：

```c
// myheader.h
#ifndef MYHEADER_H      // 如果没定义过
#define MYHEADER_H      // 就定义它

// 头文件内容

#endif                  // 结束
```

## 10. 编译与链接：再往前走一步

前面说了四个阶段。这里展开讲链接，因为在有 Python 背景的人看来，这是整个流程里最不直观的一环。

### 声明 vs 定义

```c
// 声明：告诉编译器"这个东西存在"
extern int global_var;     // 有一个全局变量，但不在这
void some_func(int x);     // 有一个函数长这样，但不在这

// 定义：实际分配内存 / 生成代码
int global_var = 42;
void some_func(int x) {
    printf("%d\n", x);
}
```

声明**可以出现多次**（只要一致），定义**只能出现一次**（违反就是链接错误 "multiple definition"）。

### 多文件编译

```bash
# 每个 .c 单独编译成 .o
gcc -c main.c -o main.o        # -c 表示只编译，不链接
gcc -c helper.c -o helper.o

# 链接所有 .o 成可执行文件
gcc main.o helper.o -o program
```

为什么要分开？改一个文件只重编译那个文件，不用重新编译整个项目。项目越大意义越大。

### make 和 Makefile

`make` 靠时间戳判断哪些文件需要重编译：目标比依赖旧，就重新生成。

```makefile
program: main.o helper.o
	gcc main.o helper.o -o program

main.o: main.c helper.h
	gcc -c main.c -o main.o

helper.o: helper.c helper.h
	gcc -c helper.c -o helper.o

clean:
	rm -f *.o program
```

CMake 是 make 的上层工具，生成 Makefile（或 Ninja 文件）。项目复杂后自然会用到，现在知道就好。

### 静态库 vs 动态库

- **静态库** (`.a` Linux / `.lib` Windows)：链接时把库代码复制进可执行文件。文件大，但不依赖外部环境。
- **动态库** (`.so` Linux / `.dll` Windows)：运行时动态加载。文件小，升级不需要重编译，但部署时要带库文件。

## 11. 常见陷阱速查

| 陷阱 | 症状 | 原因 | 预防 |
|------|------|------|------|
| 野指针 | 随机崩溃 | 指针未初始化 | 声明时赋 NULL |
| 悬垂指针 | 奇怪的错误值 | free 后继续用 | free 后赋 NULL |
| 内存泄漏 | 内存占用越来越大 | 忘了 free | 谁 malloc 谁 free |
| 缓冲区溢出 | 覆盖相邻变量/崩溃 | 数组越界写入 | 检查边界，用 `strncpy` 替代 `strcpy` |
| 栈溢出 | 程序崩溃 | 栈上放了太大的数组 / 递归太深 | 大数据用堆分配 |
| 整数溢出 | 结果不对 | `int` 超出范围 | 检查范围，用 `long long` |
| 有符号/无符号混用 | 比较结果反常 | `-1 > 0U` 为真 | 避免混用，编译器加 `-Wsign-compare` |
| 分号误用 | 逻辑错误或编译失败 | `if (x); foo();` | 编译警告全开 |

## 12. 与嵌入式的接口预览

你选的方向是嵌入式，这里列出 C 语言里哪些部分会直接对接嵌入式开发，心里有个地图。

| C 概念 | 嵌入式场景 |
|--------|-----------|
| `volatile` | 硬件寄存器的值随时可能被硬件改变，禁止编译器优化 |
| 位运算 | 操作寄存器的特定位（开/关 GPIO 引脚、配置定时器模式） |
| 函数指针 | 中断向量表——中断发生时自动跳转到对应处理函数 |
| `const` | 常量数据放进 Flash 而不是 RAM（MCU 的 RAM 很小） |
| `static` | 限制函数/变量的可见范围，避免命名冲突 |
| struct + 位域 | 把硬件寄存器按位拆分，直接映射到结构体成员 |
| 动态内存谨慎使用 | 嵌入式里 `malloc` 可能不允许（内存碎片、不确定性） |
| 没有操作系统 | `main()` 永远不会返回，主循环 `while(1)` 跑一辈子 |

## 学习建议

针对你的情况（Python 熟练，C 遗忘但有底子）：

- **前 3 天**：快速过一遍基本语法（类型、控制流、函数），这是恢复性训练。重点看和 Python 不一样的地方——类型声明、分号、花括号、`return` 的语义。
- **核心攻坚**：指针和动态内存。练到你看到一个指针声明能立刻画出内存图。多写几个链表操作，反复体会 `->`、`*`、`&` 的区别。
- **随手编译**：每个概念都写 5 行代码编译跑一下。不编译你感觉不到 UB 的存在。
- **打开所有警告**：`gcc -Wall -Wextra -pedantic`。把警告当错误处理：`-Werror`。

# Claude Code Skills

本目录包含三个自建的 Claude Code 技能（Skills）。将对应目录复制到 `~/.claude/skills/` 即可在 Claude Code 中使用。

## zjk-mastery

技术知识精通学习工作流。三阶段闭环：输入（内容发现 + AI 笔记生成）→ 思考（互动问答 + 个人笔记 + 模拟题）→ 输出（A4 默写 + 掌握度测试）。支持间隔重复回测（周/月/年）。

> AI 笔记的写作部分参考了 [ljg-paper](https://github.com/lijigang/ljg-skills/tree/master/skills/ljg-paper) skill 的 AI 写作模块，对写作红线、写作原则和笔记模板进行了优化。

### 待优化：内容发现阶段的资源搜索质量

**问题：** 当前模型在拿到一个学习主题时，容易找不到优质资源。具体表现为：
1. 搜索 query 太泛，一次搜索就定结果，缺乏对不同类型主题的差异化搜索策略
2. 搜索工具不可靠时（空结果、限流），缺乏应对策略，只会换 query 硬搜
3. 对新造词/新范式类主题，搜索入口与成熟技术不同，但没有区分处理

**根因分析：** SKILL.md 的 1.2 节只写了"用 WebSearch 搜索"加一个优先级列表（原始论文 > 官方文档 > 高质量博客 > 经典教程），但缺少：
- 如何构造有效的搜索 query
- 针对不同类型主题的差异化搜索策略
- 条件触发的渐进式搜索流程（而非固定轮次）
- 搜索失败时的应对策略
- 资源质量的验证手段

**优化方案（对 SKILL.md 1.2 节的改造）：**

#### 1. 按主题类型选择搜索策略

不是所有主题都用同一套搜索方式。拿到主题后先判断类型，再选策略：

| 主题类型 | 判断依据 | 核心搜索目标 |
|----------|----------|--------------|
| 框架/库 | 有明确的名称和版本（如 LangChain、asyncio） | 官方文档 > GitHub 仓库 > 快速上手教程 |
| 概念/范式 | 描述一种思想或方法论（如 RAG、ReAct） | 原始论文/博客 > 权威解读 > 对比分析 |
| 新造词/新范式 | 近 1-2 年出现的新术语，搜索结果少于 3 条有效命中（如 Harness Engineering、Vibe Coding） | 找提出者和原始发布平台 > 社区解读 > 对比分析 |
| 算法/模型 | 有数学公式或模型架构（如 Transformer、YOLO） | 原始论文 > 可视化讲解 > 代码实现 |
| 架构/工程 | 描述系统设计或工程实践（如微服务、消息队列） | 实战案例 > 生产环境 postmortem > 架构对比 |

**新造词/新范式的特殊处理：**
这类主题没有 "official documentation"，搜索 `[主题名] official documentation` 是无效的。有效的搜索方式是找"谁在什么时候在哪提出的"：
- `[主题名] who proposed` / `[主题名] coined by`
- `[提出者/机构名] [主题名]`（如 `OpenAI harness engineering`、`Karpathy vibe coding`）
- `[主题名] blog post` / `[主题名] announcement`
- 如果能确定提出者，直接尝试猜测其博客/官网 URL（如提出者是 OpenAI → `openai.com/index/[主题名]/`）

#### 2. 条件触发的渐进式搜索

不要固定跑 3 轮搜索。每一轮结束后判断是否需要继续：

**第一轮：定位核心资源（必须执行）**
- 构造 2 个 query，目标是找到主题的"第一来源"
- query 模板（按主题类型选择）：
  - 框架/库：`[主题名] official documentation`、`[主题名] GitHub`
  - 概念/范式：`[主题名] original paper`、`[作者名/机构名] [主题名]`
  - 新造词：`[主题名] who proposed`、`[提出者/机构名] [主题名]`
  - 算法/模型：`[主题名] paper`、`[主题名] explained`
  - 架构/工程：`[主题名] best practices`、`[主题名] real world`

**第一轮结束后的判断：**
- ✅ 找到 ≥2 个有效资源 → 进入第二轮补充深度内容
- ⚠️ 只找到 1 个或质量不确定 → 先用这一个，同时进入第二轮
- ❌ 没有找到有效资源 → **跳过第二轮，进入搜索失败应对流程**（见下方）

**第二轮：补充深度内容（条件触发）**
- 用第一轮找到的资源中的关键词，搜更深入的解读
- query 模板：
  - `[主题名] tutorial` / `[主题名] explained` / `[主题名] deep dive`
  - `[主题名] best practices` / `[主题名] common mistakes`
  - `[主题名] site:medium.com` 或 `site:dev.to`（限定高质量社区）
  - `[主题名] 知乎` / `[主题名] 掘金`（中文场景）

**第三轮：查缺补漏（按需）**
- 仅在前两轮覆盖不全时执行
- query 模板：
  - `[主题名] vs [竞品/替代方案]`（对比类）
  - `[主题名] production` / `[主题名] real world`（实战类）
  - `[主题名] interview questions`（面试导向时）

#### 3. 搜索失败应对流程

当第一轮搜索没有找到有效资源时，不要继续换 query 硬搜。按以下顺序处理：

**第一步：主动问用户（优先）**
- 告诉用户搜索结果不理想，请用户提供线索
- 提示模板："这个主题我没有找到可靠来源。你是否知道它是在哪发布的（博客、论文、推文）？或者有相关链接？"
- 用户提供的材料优先级高于搜索结果

**第二步：猜测直接 URL（补充）**
- 如果能判断提出者/来源机构，尝试直接访问可能存在的 URL
- 常见模式：`[公司名].com/index/[主题名]/`、`[作者名].github.io/[主题名]/`
- 猜测的 URL 必须通过 WebFetch 验证内容确实存在

**第三步：终止并等待用户**
- 如果以上都失败，明确告知用户无法找到可靠来源，暂停内容发现阶段
- 对于不确定的内容直接说不知道，不要瞎编——对于追求知识精通的场景，带有幻觉的笔记比没有笔记更有害
- 等待用户补充材料或提供新的线索后再继续

#### 4. 资源质量验证

找到资源后，不要直接推荐。做两步验证：

**第一步：URL 过滤（不用 fetch，看 URL 就能判断）**
- 优先域名：官方域名 > github.com > 长期维护的技术博客
- 排除域名：低质量聚合站、SEO 农场、纯转载站
- 看日期：技术资源超过 2 年的需要标注"可能过时"

**第二步：抽样验证（fetch 后快速判断）**
- 仅对排名前 2 的候选资源做 WebFetch（控制调用次数）
- 判断标准：有实质性内容（不是"什么是 X"的泛泛介绍）、结构清晰、有代码示例或图表

#### 5. 输出格式优化

把搜索结果分两层呈现给用户：

**第一层：推荐列表（3-5 个精选）**
```
| # | 资源 | 类型 | 来源质量 | 为什么选它 |
```

**第二层：备选列表（额外 2-3 个）**
- 用一句话说明为什么没放进推荐（如"内容不错但较旧"、"偏入门可以跳过"）

#### 6. 用户确认环节

当前 skill 已有让用户确认的步骤，保持不变。但在呈现时增加：
- 每个推荐资源附一句"推荐理由"（不是"权威性高"这种空话，而是"这篇用动画演示了 attention 的计算过程"）
- 如果用户提供了自己的材料，优先使用用户材料，搜索结果作为补充

**实施范围：** 仅修改 `skills/zjk-mastery/SKILL.md` 的 1.2 节，将上述策略写入 skill prompt。其他阶段和文件不动。

## paper-reading

吴恩达式论文精读助手。引导用户逐节阅读学术论文，通过提问和纠正帮助理解核心概念，自动搜索补充资料，生成学习笔记，支持保存/恢复学习进度。
